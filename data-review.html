Manufacturing AX Agent

CNC 설비의 예지보전 + 온톨로지 + LLM 자율 판단 관제 시스템

배경 및 문제 정의

제조 현장의 설비 데이터(SCADA), 생산 맥락(MES), 자원 상태(ERP), 정비 지식(매뉴얼)이
각각 독립적인 시스템(Silo)에 분산되어 저장 및 관리되고 있습니다.

설비 이상 발생 시, 작업자가 3~4개의 시스템을 수동으로 순회하며 상황을 대조하고 판단해야 합니다.
이로 인해 의사결정이 지연되고, 설비의 평균 수리 시간(MTTR)과 비가동 시간(Downtime)이
불필요하게 증가하여 생산 원가 손실이 발생합니다.

프로젝트 목표

기존의 분절된 IT/OT 데이터를 데이터베이스 스키마 단위에서 통합하고,
센서 수치 예측(Forecasting)과 LLM(대형 언어 모델)을 결합하여
설비 정지 전 선제적 알람과 비즈니스 맥락이 반영된 자율 조치 지시서를 즉각적으로 제공하는
AI 에이전트 시스템을 구축합니다.

프로젝트 범위 (Scope)

본 프로젝트는 데이터가 도착한 이후의 분석/판단/관제 레이어(F1~F6)를 구축하는 것이 목표입니다.
실제 장비에 IoT 센서를 부착하거나, FOCAS/OPC UA 같은 제조사 폐쇄 프로토콜을 연결하는
데이터 수집 레이어는 범위 밖입니다.

범위 안 :
센서 데이터 전처리(F1) → 이상탐지/예측(F2) → IT/OT 동기화(F3) → GraphRAG 검색(F4) → LLM 자율 판단(F5) → 대시보드(F6)

범위 밖 :
IoT 센서 부착, 프로토콜 브릿지(FOCAS, OPC UA), 엣지 게이트웨이, 공장 네트워크 접근

수집된 데이터를 온톨로지 + GraphRAG + LLM으로 자율 판단까지 연결하는 것이 차별화 포인트입니다.

설계 원칙 (5가지)

  • 원칙 1 :  모든 데이터는 equipment_id와 timestamp 기준으로 연결
  • 원칙 2 :  센서만으로 가능한 기능 vs 이벤트 필요 기능 구분
  • 원칙 3 :  온톨로지는 Entity/Relation 먼저 정의
  • 원칙 4 :  합성 데이터는 LLM 판단이 달라지는 시나리오 설계
  • 원칙 5 :  내부 표준 형식(Canonical Model) 기준 설계 — 원본 데이터 형식이 아닌
                 내부 표준 형식으로 정규화. 향후 고객사 확장 시 Onboarding Layer 추가만으로 대응 가능

제약사항

각 제약의 상세 배경과 검토된 대안은 ADR 페이지에서 확인할 수 있습니다.

  • CNC 단일 도메인 — 도메인 혼합 시 조인 키 충돌 및 LLM 환각 방지 (ADR-000)
  • RUL 대신 Forecasting — 합성 타겟 변수로는 모델 신뢰성 평가 불가 (ADR-001)
  • 합성 MES/ERP — 기업 데이터 비공개, Kaggle 타임라인에 맞춰 합성 (ADR-002)
  • DB 2개로 제한 — 1인 개발, PostgreSQL 확장으로 3역할 통합 (ADR-003)
  • 폴링 방식 — Kafka/Flink 없이 5초 주기 조회로 실시간 모사 (ADR-004)
  • 데이터 수집 레이어 범위 밖 — 실제 장비/IoT/프로토콜 연결 없이 공공 데이터 + 합성으로 대체

Q. 왜 RUL(잔여수명)이 아닌 Forecasting인가? (ADR-001)

RUL은 "이 부품이 몇 시간 뒤에 고장난다"를 직접 예측하지만,
신뢰할 수 있는 타겟 변수를 합성하기 어렵습니다.
대신 실제 센서값의 미래를 예측하고, 임계치 돌파 여부로 판단하는 Forecasting을 택했습니다.

예지보전을 구성하는 5가지 데이터

"예지보전 데이터"라는 단일 테이블이 있는 것이 아닙니다. 아래 데이터가 합쳐져서 만들어집니다.

  • 센서 신호 — 전류, 전압, 전력, 속도, 가속도 → 이상탐지, Forecasting
  • 이벤트 근거 — 알람 발생, 정비 시점, 고장 시점 → 전조와 실제 이벤트 연결
  • 운영 맥락 — 작업지시, 납기 긴급도, 생산 잔량 → 멈출지/감속할지 판단
  • 자원 맥락 — 부품 재고, 리드타임, 교체 이력 → 실제 조치 가능성 판단
  • 정비 지식 — 고장 코드, 조치 절차, 매뉴얼 → GraphRAG 검색 근거

F1 실시간 시계열 데이터 수집 및 전처리

원시 센서 데이터(CNC Mill)를 수집하여 결측치 및 이상치를 통계적으로 제거합니다.
Sliding Window 크기로 이동 평균, 분산 등 파생 변수(Feature)를 추출합니다.

  • 입력 :  CNC 센서 CSV (48컬럼, 100ms 샘플링)
  • 출력 :  깨끗한 시계열 데이터 (정규화, 결측 처리, 파생 변수)
  • 데이터 :  Kaggle CNC Mill (핵심) + Bosch CNC (보조, 진동)
F2 사전 위험 예측 및 이상 탐지

비지도 학습 모델로 Anomaly Score를 판별하고,
LSTM 등 시계열 모델로 미래 센서값을 예측하여 임계치 돌파 여부를 선제 알림합니다.

  • 입력 :  F1 전처리 출력 + 공구 상태 라벨(worn/unworn)
  • 출력 :  이상 점수(Anomaly Score) + 미래 예측값
  • 데이터 :  F1 출력 + 정비/고장 이벤트 로그(Phase 2 합성)
F3 IT/OT 데이터 타임라인 동기화

F2 알람 시점 기준으로 MES 작업지시(납기/우선순위)와
ERP 부품재고(재고량/리드타임)를 조회하여 단일 JSON으로 병합합니다.

  • 입력 :  F1 출력(OT) + MES(IT) + ERP(IT)
  • 출력 :  통합 운영 맥락 JSON
  • 데이터 :  합성 생성 (ADR-002)
F4 지식그래프 하이브리드 검색 (GraphRAG)

Neo4j 온톨로지에서 '설비 → 고장 코드 → 부품 → 매뉴얼' 관계망을 순회합니다.
BM25(키워드) + Vector Search(증상 설명)를 결합한 하이브리드 검색입니다.

  • 입력 :  Neo4j 온톨로지 + 정비 매뉴얼 텍스트
  • 출력 :  관련 매뉴얼, 부품, 고장 이력 검색 결과
  • 데이터 :  온톨로지(직접 설계) + 매뉴얼(수집+합성) + 고장코드(직접 설계)
F5 LLM 기반 자율 조치 판단

F2(이상 점수) + F3(운영 맥락) + F4(정비 지식)의 결과를 LLM에 넘겨,
비즈니스 맥락이 반영된 조치 보고서를 자동 작성합니다.

  • 입력 :  F2 + F3 + F4 출력 조합
  • 출력 :  자율 조치 지시서 (보고서)
F6 대시보드 (관제 화면)

F1~F5의 모든 결과를 한 화면에 시각화합니다.
ADR-005에 의해 Phase 1부터 조기 착수, 기능 추가 시마다 시각적으로 검증합니다.

  • 입력 :  F1~F5 전체 출력
  • 출력 :  실시간 관제 대시보드

"유효 42컬럼"의 의미

원래 48개 컬럼 중 6개가 분석에 쓸모없습니다.

상수 컬럼 2개: S1_SystemInertia(항상 12.0), M1_CURRENT_PROGRAM_NUMBER(항상 1.0)

Z축 전기 피드백 4개: CurrentFeedback, DCBusVoltage,
OutputCurrent, OutputVoltage — 18개 실험 전부에서 값이 0.
센서 미연결 또는 미수집.

(단, Z축의 위치/속도/가속도는 정상입니다! 전기 데이터만 0이에요.)

48 - 2(상수) - 4(Z축 전기) = 42개가 실제 분석에 쓸 수 있는 컬럼.

합성 데이터 의심: Multi-Sensor는 센서값이 너무 균등한 분포를 보이고
마모량에 음수 값이 존재합니다. Milling RUL은 진동이 표준정규분포이고
RUL이 선형 감소합니다. 실제 센서 데이터가 아닌 인위적으로 생성된 데이터일 가능성이 높습니다.
Milling RUL의 RUL 컬럼은 ADR-001에 의해 미사용하며, 센서 패턴만 참고합니다.

미확보 데이터 전략

Phase 1에서 해결 — 합성 규칙 설계

  • timestamp :  sequence × 100ms → datetime 변환. 18개 실험을 시간축으로 정렬 가능하게 합성
  • equipment_id :  experiment_01~18 → CNC-001~018 매핑. 설비 단위 분석의 기준 키

규칙 초안 완료 (데이터 검증 후 확정)

Phase 2에서 해결 — Python 합성 생성 (ADR-002)

  • MES 작업지시 :  work_order_id, due_date, priority, status 등. 센서 timestamp와 정렬되어야 하며, LLM 판단이 달라질 만큼 비즈니스 차이를 반영
  • ERP 부품 재고 :  part_id, stock_quantity, lead_time_days 등. 온톨로지 부품 정보와 일치해야 하며, 재고 충분/부족 시나리오 모두 포함
  • 정비/고장 이벤트 로그 :  failure_code, maintenance_time 등. 센서 이상 시점과 실제 고장 사이의 연결고리

비즈니스 시나리오 4종(S1~S4) 초안 완료 — 긴급/여유 납기 × 재고 충분/부족 조합

Phase 3에서 해결 — 직접 설계 + 수집

  • 온톨로지 관계 스키마 :  설비 → 고장 코드 → 부품 → 매뉴얼 관계를 Neo4j에 직접 설계. CNC 단일 도메인이므로 최소 범위로 시작
  • 고장 코드 체계 :  E-001 = 스핀들 과열처럼 코드와 의미를 정의. 산업 표준 참고하되 프로젝트 범위에 맞게 축소 설계
  • 정비 매뉴얼 텍스트 :  CNC 제조사 공개 기술 문서 + 합성 보강. GraphRAG 벡터 검색의 텍스트 근거

확보 방안 미확정 (Open Items #2, #3) — 대량 수집보다 작은 범위의 정확한 연결이 우선

대체 설계 — 센서 한계 보완

  • 온도 센서 :  Kaggle CNC Mill에 미포함. 전류·전력 변화량으로 간접 추정 설계 (전류 급증 → 발열 의심)
  • 진동 센서 :  Kaggle CNC Mill에 미포함. Bosch CNC(2kHz 진동)로 보완, Multi-Sensor CNC는 패턴 참고만
EDA : 데이터 탐색 분석

Kaggle CNC Mill 데이터를 6가지 관점에서 시각적으로 탐색합니다.

애자일 목표

이 프로젝트에서 애자일이 무엇이고, 왜 이 방식을 선택했으며, 어떻게 실천하는지 정리합니다.

왜 애자일인가

1인 개발 + 입문자 수준 + 불확실한 기술 스택이라는 현실이 있습니다.

온톨로지, GraphRAG, LLM 자율 판단 같은 기술을 처음 다루기 때문에 처음부터 완벽한 설계를 할 수 없습니다. 대신 짧은 Phase 단위로 학습하고, 검증하고, 다음으로 넘기는 방식이 가장 현실적입니다.

  • 리소스 한계 :  1인이 운영 가능한 최소한으로 결정 (예 : DB 2개로 통합 — ADR-003)
  • 학습 병행 :  완벽한 이해 없이 시작하되, 단계별로 구체화 (예 : 온톨로지는 Phase 0에서 원칙만, Phase 3에서 스키마)
  • 조기 검증 :  끝까지 기다리지 않고 중간 결과물로 방향을 확인 (예 : 대시보드 MVP 앞당김 — ADR-005)

이 프로젝트의 애자일 사이클

STEP 1
가설과 범위를 먼저 고정
이번 단계에서 무엇을 결정할지, 무엇은 아직 미결로 둘지 먼저 나눕니다. 과한 선행 구현을 막습니다.
STEP 2
실데이터와 실제 제약으로 검증
문서 가설이 실제 데이터와 맞는지 대조합니다. 안 맞는 부분은 갭으로 기록한 뒤 수정합니다.
STEP 3
결정은 ADR로 남김
왜 이 방향을 택했는지 근거와 대안을 기록합니다. 다음 단계에서 흔들리지 않기 위해서입니다.
STEP 4
산출물을 다음 단계 입력으로
이번 단계의 산출물이 바로 다음 단계의 입력이 됩니다. Phase가 순차적으로 연결됩니다.

실천 원칙 6가지

01 짧은 Phase, 빠른 사이클

Phase 0은 3일 만에 완주했습니다. 완벽하지 않아도 다음 단계로 이동하고, 미결 사항은 Open Items로 명시적으로 관리합니다.

02 의사결정은 ADR로 기록

"결정했다"만이 아니라 문제 상황 → 검토 대안 → 선택 이유 → 제약사항을 모두 남깁니다. 나중에 수정할 때 맥락이 있어야 하기 때문입니다.

03 갭 분석으로 가정을 검증

문서에 쓴 가정을 실제 데이터와 대조합니다. Phase 0에서만 3회 갭 분석을 실시하여 timestamp 부재, 진동 센서 미포함 등을 발견했습니다.

04 제약을 제약으로 받아들임

"최선"이 아니라 "지금 가능한 것"으로 결정합니다. Kafka 대신 5초 폴링(ADR-004), 다중 DB 대신 PostgreSQL 통합(ADR-003)이 그 예입니다.

05 조기 시각화

파이프라인 완성을 기다리지 않고 대시보드 MVP를 앞당겼습니다(ADR-005). 부분적 결과물이라도 눈에 보여야 방향이 맞는지 판단할 수 있습니다.

06 살아있는 문서

모든 문서는 일회성이 아닙니다. Phase가 진행되면 PRD, Open Items, README가 함께 업데이트됩니다. 문서와 코드의 괴리를 허용하지 않습니다.

문서와 구현의 분리

이 프로젝트는 단계별로 중심 활동이 다릅니다.

  • Phase 0 (문서 중심) :  PRD, 데이터 전략, ADR, 제약사항 등 의사결정 기록. 코드 없음
  • Phase 1 (분석 중심) :  EDA 노트북으로 실데이터 검증 + 새로운 ADR 작성
  • Phase 2 (설계 중심) :  DB 스키마, 파이프라인, 합성 데이터 모델 구체화
  • Phase 3+ (구현 중심) :  FastAPI, PostgreSQL, Neo4j, 대시보드 개발. 코드 규칙 적용

코드 규칙(타입 힌트, 한국어 주석, Docker Compose 등)은 Phase 2 이후부터 적용됩니다.

의사결정과 미결 관리 구조

ADR (Architecture Decision Record)

기술 의사결정마다 "Context → Options → Decision → Reasoning → Consequences" 구조로 기록합니다. 현재까지 6건(ADR-000~005)이 확정되었고, Phase가 진행되면서 추가됩니다.

Open Items (미결 사항)

지금 결정할 수 없는 것은 Open Items로 명시적으로 분류합니다. 각 항목마다 "왜 미결인가"와 "언제 결정할 것인가"를 기록합니다.

  • A. 데이터 관련 :  Phase 1 EDA에서 결정 (timestamp 규칙, 매뉴얼 확보 등)
  • B. 모델링 :  Phase 1~2에서 결정 (이상탐지 모델, LSTM 윈도우 등)
  • C. 시스템 :  Phase 2에서 결정 (LLM 선택, 프론트엔드 전략 등)
  • D. 관리 :  Phase 2 완료 후 정의 (장기 마일스톤)

        ADR-000
        CNC 단일 도메인
        Phase 0
        결정

        CNC 밀링 단일 도메인으로 통일합니다.

        검토된 대안

        Option A :  NASA + CNC + 반도체 등 다중 도메인 혼합
        Option B :  CNC 밀링 단일 도메인 (선택)

        선택 근거

        다중 도메인 혼합 시 데이터 조인 키 충돌 및 LLM 환각이 발생합니다.
        하나의 도메인에서 깊게 파는 것이 MVP에 적합합니다.

        이로 인한 제약

        CNC 밀링 외 도메인(터빈, 반도체 등)은 MVP 범위에서 제외됩니다.

        ADR-001
        RUL 포기, Forecasting 방식 채택
        Phase 0
        결정

        잔여수명(RUL) 예측 대신, 센서값의 미래를 예측(Forecasting)하고
        임계치 돌파 여부로 판단하는 방식을 택합니다.

        검토된 대안

        Option A :  RUL 기반 — "남은 수명 몇 시간" 직접 예측
        Option B :  Forecasting 기반 — 센서값 미래 예측 후 임계치 판단 (선택)

        선택 근거

        공개 데이터에 실제 고장 시점이 없어 가상의 RUL을 합성해야 하는데,
        합성 타겟으로 학습한 모델의 신뢰성을 평가할 방법이 없습니다.
        실제 센서값의 미래를 예측하는 방식이 검증 가능하고 더 신뢰할 수 있습니다.

        이로 인한 제약

        "설비 수명이 몇 시간 남았습니다" 같은 직관적 출력은 불가합니다.
        대신 "30분 뒤 전류가 임계치를 넘을 가능성 높음" 형태로 제공합니다.

        ADR-002
        MES/ERP 합성 데이터 생성
        Phase 0
        결정

        MES(작업지시)와 ERP(부품재고) 데이터를 시나리오 기반으로 합성합니다.

        검토된 대안

        Option A :  실제 기업 데이터 확보 시도
        Option B :  시나리오 기반 합성 (선택)

        선택 근거

        기업 IT 데이터는 비공개이며 확보가 현실적으로 불가능합니다.
        Kaggle 센서 타임라인에 맞춰 합성하면 F3 IT/OT 동기화 파이프라인을 구현할 수 있습니다.

        이로 인한 제약

        합성 데이터이므로 실제 MES/ERP의 복잡한 비즈니스 로직은 반영하지 못합니다.

        결정

        PostgreSQL에 TimescaleDB(시계열) + pgvector(벡터) 확장으로 3역할 통합하고,
        온톨로지는 Neo4j를 별도로 사용합니다. 총 DB 2개.

        검토된 대안

        Option A :  역할별 DB 분리 (TimescaleDB + Weaviate + PostgreSQL + Neo4j = 4개)
        Option B :  PostgreSQL 확장 통합 + Neo4j (선택)

        선택 근거

        1인 개발 환경에서 DB 4개를 운영하는 것은 비현실적입니다.
        PostgreSQL 확장으로 시계열/벡터/관계형을 하나로 통합하면 운영 부담이 크게 줄어듭니다.

        이로 인한 제약

        전용 시계열 DB(InfluxDB)나 전용 벡터 DB(Weaviate)에 비해 성능은 낮을 수 있습니다.
        MVP 규모에서는 충분하며, 필요 시 Phase 3 이후 분리를 검토합니다.

        Phase 0

        검토된 대안

        Option A :  Kafka + Flink 스트리밍 파이프라인
        Option B :  5초 주기 폴링 (선택)

        선택 근거

        실시간 관제 시나리오를 충분히 구현할 수 있습니다.

        이로 인한 제약

        밀리초 단위의 실시간 반응은 불가합니다. 5초 지연은 허용 범위입니다.
