# Key Constraints (핵심 제약사항)

> ADR 의사결정에서 도출된 프로젝트 핵심 제약사항입니다.
> 새로운 기능이나 설계를 검토할 때 이 제약을 먼저 확인합니다.

---

| 제약 | 결정 근거 | 관련 ADR |
|------|----------|---------|
| CNC 단일 도메인 | 도메인 혼합 시 데이터 조인 키 충돌 및 LLM 환각 방지 | ADR-000 |
| RUL 대신 Forecasting | 합성 타겟 변수로는 모델 신뢰성 평가 불가 | ADR-001 |
| 합성 MES/ERP | 기업 데이터 비공개, 기준 센서 타임라인에 맞춰 합성 | ADR-002 |
| DB 2개로 제한 | 1인 개발, PostgreSQL 확장으로 3역할 통합 (시계열+관계형+벡터) | ADR-003 |
| 폴링 방식 | Kafka/Flink 없이 5초 주기 조회로 실시간 모사 | ADR-004 |
| 대시보드 조기 개발 | 파이프라인과 병행하여 시각적 검증 가능하게 | ADR-005 |
| 데이터 수집 레이어는 범위 밖 | 실제 장비/IoT 센서/프로토콜 연결 없이, 공공 데이터 + 합성으로 대체 | — |

## 프로젝트 범위 (Scope)

본 프로젝트는 **데이터가 도착한 이후의 분석/판단/관제 레이어(F1~F6)**를 구축하는 것이 목표입니다.

```
[현장 장비] → [IoT/프로토콜 수집] → [F1] → [F2] → [F3] → [F4] → [F5] → [F6]
              ─────────────────    ──────────────────────────────────────────
              범위 밖 (수집 레이어)    범위 안 (분석/판단/관제 레이어)
```

### 범위 밖으로 둔 이유

실제 제조 현장에서 센서 데이터를 수집하려면 다음이 필요합니다:

- **IoT 센서 부착**: 구형 CNC에는 진동/온도 센서가 기본 탑재되어 있지 않아 별도 부착 필요
- **프로토콜 연결**: 화낙(FOCAS), 지멘스(OPC UA) 등 제조사별 폐쇄 프로토콜 해결 필요
- **엣지 게이트웨이**: 공장 내 데이터 수집→전처리→전송 장비 필요
- **보안/현장 접근**: 공장 네트워크 접근 권한, 장비 정지 리스크 관리

이 영역은 실제 장비와 현장 접근이 전제되어야 하며, 1인 개발 프로젝트에서는 현실적으로 불가능합니다.
현재 이 문제를 전문으로 해결하는 IIoT 스타트업들이 별도 사업 영역으로 다루고 있는 수준입니다.

### 우리의 차별화 포인트

수집된 데이터를 **온톨로지 + GraphRAG + LLM으로 자율 판단까지 연결하는 시스템**을 구축하는 것.
데이터 수집은 하드웨어와 현장 작업으로 해결하는 영역이고, 수집 후 **무엇을 할 수 있는지**가 진짜 차별화입니다.

### 향후 확장 시

고객사에 실제 적용할 때는 수집 레이어(IoT 센서 + 프로토콜 브릿지 + 엣지)를 앞단에 추가하면 됩니다.
이 부분은 [customer-onboarding-ontology-strategy.md](../future/customer-onboarding-ontology-strategy.md)에서 다룹니다.

## 설계 원칙

data-collection-strategy.md에서 정의한 5가지 핵심 원칙:

1. 모든 데이터는 `equipment_id`와 `timestamp` 기준으로 연결
2. 센서만으로 가능한 기능 vs 이벤트 필요 기능 구분
3. 온톨로지는 Entity/Relation 먼저 정의
4. 합성 데이터는 LLM 판단이 달라지는 시나리오 설계
5. **내부 표준 형식(Canonical Model)** 기준 설계 (Phase 0에서는 원칙만, 구체 스키마는 Phase 1-2)
