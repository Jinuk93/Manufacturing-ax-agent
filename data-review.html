Manufacturing AX Agent

CNC 설비의 예지보전 + 온톨로지 + LLM 자율 판단 관제 시스템

배경 및 문제 정의

제조 현장의 설비 데이터(SCADA), 생산 맥락(MES), 자원 상태(ERP), 정비 지식(매뉴얼)이
각각 독립적인 시스템(Silo)에 분산되어 저장 및 관리되고 있습니다.

설비 이상 발생 시, 작업자가 3~4개의 시스템을 수동으로 순회하며 상황을 대조하고 판단해야 합니다.
이로 인해 의사결정이 지연되고, 설비의 평균 수리 시간(MTTR)과 비가동 시간(Downtime)이
불필요하게 증가하여 생산 원가 손실이 발생합니다.

프로젝트 목표

기존의 분절된 IT/OT 데이터를 데이터베이스 스키마 단위에서 통합하고,
센서 수치 예측(Forecasting)과 LLM(대형 언어 모델)을 결합하여
설비 정지 전 선제적 알람과 비즈니스 맥락이 반영된 자율 조치 지시서를 즉각적으로 제공하는
AI 에이전트 시스템을 구축합니다.

프로젝트 범위 (Scope)

본 프로젝트는 데이터가 도착한 이후의 분석/판단/관제 레이어(F1~F6)를 구축하는 것이 목표입니다.
실제 장비에 IoT 센서를 부착하거나, FOCAS/OPC UA 같은 제조사 폐쇄 프로토콜을 연결하는
데이터 수집 레이어는 범위 밖입니다.

범위 안 :
센서 데이터 전처리(F1) → 이상탐지/예측(F2) → IT/OT 동기화(F3) → GraphRAG 검색(F4) → LLM 자율 판단(F5) → 대시보드(F6)

범위 밖 :
IoT 센서 부착, 프로토콜 브릿지(FOCAS, OPC UA), 엣지 게이트웨이, 공장 네트워크 접근

수집된 데이터를 온톨로지 + GraphRAG + LLM으로 자율 판단까지 연결하는 것이 차별화 포인트입니다.

제약사항

각 제약의 상세 배경과 검토된 대안은 ADR 페이지에서 확인할 수 있습니다.

  • CNC 단일 도메인 — 도메인 혼합 시 조인 키 충돌 및 LLM 환각 방지 (ADR-000)
  • RUL 대신 Forecasting — 합성 타겟 변수로는 모델 신뢰성 평가 불가 (ADR-001)
  • 합성 MES/ERP — 기업 데이터 비공개, Kaggle 타임라인에 맞춰 합성 (ADR-002)
  • DB 2개로 제한 — 1인 개발, PostgreSQL 확장으로 3역할 통합 (ADR-003)
  • 폴링 방식 — Kafka/Flink 없이 5초 주기 조회로 실시간 모사 (ADR-004)
  • 데이터 수집 레이어 범위 밖 — 실제 장비/IoT/프로토콜 연결 없이 공공 데이터 + 합성으로 대체

Q. 왜 RUL(잔여수명)이 아닌 Forecasting인가? (ADR-001)

RUL은 "이 부품이 몇 시간 뒤에 고장난다"를 직접 예측하지만,
신뢰할 수 있는 타겟 변수를 합성하기 어렵습니다.
대신 실제 센서값의 미래를 예측하고, 임계치 돌파 여부로 판단하는 Forecasting을 택했습니다.

예지보전을 구성하는 5가지 데이터

"예지보전 데이터"라는 단일 테이블이 있는 것이 아닙니다. 아래 데이터가 합쳐져서 만들어집니다.

  • 센서 신호 — 전류, 전압, 전력, 속도, 가속도 → 이상탐지, Forecasting
  • 이벤트 근거 — 알람 발생, 정비 시점, 고장 시점 → 전조와 실제 이벤트 연결
  • 운영 맥락 — 작업지시, 납기 긴급도, 생산 잔량 → 멈출지/감속할지 판단
  • 자원 맥락 — 부품 재고, 리드타임, 교체 이력 → 실제 조치 가능성 판단
  • 정비 지식 — 고장 코드, 조치 절차, 매뉴얼 → GraphRAG 검색 근거

F4 지식그래프 하이브리드 검색 (GraphRAG)

Neo4j 온톨로지에서 '설비 → 고장 코드 → 부품 → 매뉴얼' 관계망을 순회합니다.
BM25(키워드) + Vector Search(증상 설명)를 결합한 하이브리드 검색입니다.

  • 입력 :  Neo4j 온톨로지 + 정비 매뉴얼 텍스트
  • 출력 :  관련 매뉴얼, 부품, 고장 이력 검색 결과
  • 데이터 :  온톨로지(직접 설계) + 매뉴얼(수집+합성) + 고장코드(직접 설계)

  • 출력 :  자율 조치 지시서 (보고서)

동작 원리

  1. 시뮬레이터(실제 현장이면 SCADA)가 CNC 3대의 센서 42개 값을 5초마다 읽음
  2. PostgreSQL sensor_readings 테이블에 INSERT (와이드 형식, 44컬럼)
  3. 정규화(MinMaxScaler) — 센서마다 단위가 다르므로 0~1 범위로 통일
    예: 전류는 0~15A, 전압은 0~400V → 둘 다 0~1로
  4. 슬라이딩 윈도우(30초 = 6행) — "최근 30초의 흐름"을 하나의 입력으로 묶음
    한 시점만 보면 정상인데, 30초간 계속 오르고 있으면 이상
  5. 파생 변수 생성 — 이동평균, 이동표준편차, 변화율
    원본 값 + "얼마나 흔들리는지" + "얼마나 빠르게 변하는지"
입력 : 센서 42개 × 3대 (5초마다) DB : sensor_readings (44컬럼) 출력 : 정규화 + 윈도우 처리된 텐서

이게 없으면: 설비 상태를 아예 모릅니다. 고장이 터진 뒤에야 알게 됩니다.

F2 이상탐지 + 미래 예측

왜 필요한가: 센서 42개의 숫자를 사람이 매번 해석할 수 없습니다. 기계가 자동으로 "이건 정상이 아니다"를 판단해야 합니다. 여기에 한 단계 더 나아가, "앞으로 이상해질 것인가"도 예측해야 진정한 예지보전이 됩니다.

F2는 두 가지 모델을 병행합니다:

  • Isolation Forest(이상탐지) — "지금 이상한가?" → 현재 시점 판단
  • 1D-CNN(시계열 예측, ADR-007) — "30초 뒤에 고장날까?" → 미래 위험 예측

1단계: 이상탐지 (Isolation Forest)

  1. F1이 준비한 윈도우 데이터(최근 6행, 30초)를 16개 피처로 변환
  2. Isolation Forest 모델이 "정상 패턴에서 얼마나 벗어나 있는지" 판단
  3. IF 점수(if_score) 산출 — 0(정상) ~ 1(이상)
  4. 고장 유형 분류 — 규칙 기반으로 4종 고장코드 중 하나 배정
    S1 전류 ↑ + 고속 → 과열 / X1 전류 ↓ → 마모 / 위치 편차 → 클램프 / 기타 → 냉각

2단계: 시계열 예측 (1D-CNN, ADR-007)

  1. 최근 30초(300행)의 핵심 4피처를 입력
    X1_CurrentFeedback, S1_CurrentFeedback, S1_OutputPower, M1_CURRENT_FEEDRATE
  2. 1D-CNN이 "다음 30초(300행)의 센서값"을 예측
  3. 정상 패턴으로 학습했으므로, 비정상 데이터는 예측 오차가 급증
    정상: 예측 잘 맞음(오차 작음) / 비정상: 예측 실패(오차 큼) = "예측 오차 = 이상 점수"
  4. forecast_score 산출 — 0(정상) ~ 1(고위험)

3단계: 합산 (ADR-007 가중 융합)

  1. final_score = 0.6 × if_score + 0.4 × forecast_score
  2. final_score ≥ 0.5 → is_anomaly = true → F3~F5 트리거 발동
  3. 결과를 anomaly_scores 테이블에 INSERT (if_score, forecast_score 모두 저장)

IT/OT 데이터 융합 (F3)

"같은 고장이라도 상황에 따라 대응이 다르다" — 센서(OT)와 비즈니스(IT)를 합쳐 맥락을 만듭니다.

OT와 IT가 뭔가?

OT (운영 기술)

"설비가 지금 어떤 상태인가"
센서 42개의 실시간 값 (전류, 전압, 진동 등)
SCADA가 수집, 5초마다 갱신

IT (정보 기술)

"설비가 뭘 하고 있고, 뭘 준비해야 하는가"
MES 작업지시, 정비이력, ERP 부품재고
이벤트 발생 시 기록

현실의 문제: 대부분의 공장에서 OT(SCADA)와 IT(MES/ERP)는 서로 다른 시스템, 다른 DB, 다른 부서가 관리합니다. "스핀들 과열!"이라는 알람이 울려도, "지금 이 설비가 긴급 작업 중인지"를 확인하려면 MES 시스템에 별도로 로그인해야 합니다. F3은 이 칸막이를 허물어 한 번에 합칩니다.

F3 IT/OT 동기화 동작 원리

이게 없으면: 맥락 없는 알람만 울립니다. "고장인 건 아는데, 지금 뭘 해야 하는지"를 모릅니다.

온톨로지 + GraphRAG (F4)

"이 고장을 고치려면 뭐가 필요하고, 매뉴얼에 뭐라고 적혀있는가?"를 지식 그래프에서 찾습니다.

왜 온톨로지가 필요한가?

"SPINDLE_OVERHEAT_001이 발생했다"는 사실만으로는 부족합니다. LLM이 판단하려면:

  • "이 고장을 고치려면 어떤 부품이 필요한가?" → P002 스핀들 베어링
  • "매뉴얼에 교체 절차가 있는가?" → DOC-004, DOC-005, DOC-006
  • "과거에 같은 고장을 어떻게 수리했는가?" → 평균 75분, 베어링 1세트 소모
  • "이 부품 재고가 있는가?" → ERP에서 2세트 확인

이 관계들이 미리 Neo4j 그래프에 정리되어 있어야 빠르게 찾을 수 있습니다. 이것이 온톨로지의 역할입니다.

LLM 자율 판단 (F5)

"그래서 지금 뭘 해야 하는가?" — 모든 정보를 종합하여 AI가 최종 판단을 내립니다.

왜 LLM인가?

F2(이상 점수) + F3(비즈니스 상황) + F4(매뉴얼/이력)을 종합하려면 규칙으로는 한계가 있습니다:

규칙 기반 (if-else)

"anomaly_score > 0.8이면 정지"
근데 지금 urgent 작업 중이면? 부품이 없으면? 과거에 같은 고장을 50분만에 고쳤으면? 경우의 수가 폭발

LLM (자연어 판단)

"이 정보를 종합했을 때 최선의 조치는?"
매뉴얼 + 이력 + 상황을 자연어로 이해하고 유연하게 판단

애자일 목표

이 프로젝트에서 애자일이 무엇이고, 왜 이 방식을 선택했으며, 어떻게 실천하는지 정리합니다.

왜 애자일인가

1인 개발 + 입문자 수준 + 불확실한 기술 스택이라는 현실이 있습니다.

온톨로지, GraphRAG, LLM 자율 판단 같은 기술을 처음 다루기 때문에 처음부터 완벽한 설계를 할 수 없습니다. 대신 짧은 Phase 단위로 학습하고, 검증하고, 다음으로 넘기는 방식이 가장 현실적입니다.

  • 리소스 한계 :  1인이 운영 가능한 최소한으로 결정 (예 : DB 2개로 통합 — ADR-003)
  • 학습 병행 :  완벽한 이해 없이 시작하되, 단계별로 구체화 (예 : 온톨로지는 Phase 0에서 원칙만, Phase 3에서 스키마)
  • 조기 검증 :  끝까지 기다리지 않고 중간 결과물로 방향을 확인 (예 : 대시보드 MVP 앞당김 — ADR-005)

이 프로젝트의 애자일 사이클

STEP 1
가설과 범위를 먼저 고정
이번 단계에서 무엇을 결정할지, 무엇은 아직 미결로 둘지 먼저 나눕니다. 과한 선행 구현을 막습니다.
STEP 2
실데이터와 실제 제약으로 검증
문서 가설이 실제 데이터와 맞는지 대조합니다. 안 맞는 부분은 갭으로 기록한 뒤 수정합니다.
STEP 3
결정은 ADR로 남김
왜 이 방향을 택했는지 근거와 대안을 기록합니다. 다음 단계에서 흔들리지 않기 위해서입니다.
STEP 4
산출물을 다음 단계 입력으로
이번 단계의 산출물이 바로 다음 단계의 입력이 됩니다. Phase가 순차적으로 연결됩니다.

실천 원칙 6가지

01 짧은 Phase, 빠른 사이클

Phase 0은 3일 만에 완주했습니다. 완벽하지 않아도 다음 단계로 이동하고, 미결 사항은 Open Items로 명시적으로 관리합니다.

02 의사결정은 ADR로 기록

"결정했다"만이 아니라 문제 상황 → 검토 대안 → 선택 이유 → 제약사항을 모두 남깁니다. 나중에 수정할 때 맥락이 있어야 하기 때문입니다.

03 갭 분석으로 가정을 검증

문서에 쓴 가정을 실제 데이터와 대조합니다. Phase 0에서만 3회 갭 분석을 실시하여 timestamp 부재, 진동 센서 미포함 등을 발견했습니다.

04 제약을 제약으로 받아들임

"최선"이 아니라 "지금 가능한 것"으로 결정합니다. Kafka 대신 5초 폴링(ADR-004), 다중 DB 대신 PostgreSQL 통합(ADR-003)이 그 예입니다.

05 조기 시각화

파이프라인 완성을 기다리지 않고 대시보드 MVP를 앞당겼습니다(ADR-005). 부분적 결과물이라도 눈에 보여야 방향이 맞는지 판단할 수 있습니다.

06 살아있는 문서

모든 문서는 일회성이 아닙니다. Phase가 진행되면 PRD, Open Items, README가 함께 업데이트됩니다. 문서와 코드의 괴리를 허용하지 않습니다.

문서와 구현의 분리

이 프로젝트는 단계별로 중심 활동이 다릅니다.

  • Phase 0 (문서 중심) :  PRD, 데이터 전략, ADR, 제약사항 등 의사결정 기록. 코드 없음

  • Phase 3 (구현 및 검증) :  FastAPI + F1~F6 파이프라인 + F2 Forecasting + 챗봇 + 대시보드 완성 [완료]

      Phase 2 : 온톨로지 설계

      "설비가 고장나면, 어떤 부품이 필요하고, 어떤 매뉴얼을 봐야 하는가?" — 이 질문에 기계가 스스로 답할 수 있도록 지식을 구조화합니다.

      온톨로지(Ontology)란?

      온톨로지는 "세상의 지식을 컴퓨터가 이해할 수 있는 구조로 정리하는 방법"입니다.

      예를 들어 사람은 "엔드밀이 마모되면 교체해야 한다"를 자연스럽게 알지만, 컴퓨터에게는 "엔드밀"이 무엇이고, "마모"가 무엇이며, 둘 사이에 어떤 관계가 있는지를 명시적으로 알려줘야 합니다. 이걸 정리하는 것이 온톨로지입니다.

      OWL (Web Ontology Language)

      W3C 표준의 정통 온톨로지입니다. 클래스 계층(Class Hierarchy), 속성 제약(Property Restriction), 추론 규칙(Reasoning)을 엄격하게 정의합니다.

      Protégé 같은 도구로 설계하며, "CNC_Machine은 Machine의 하위 클래스이다"처럼 논리적 추론에 강합니다.

      예: 의료 분야의 SNOMED CT, 생물학의 Gene Ontology

      Property Graph (Neo4j) ← 우리의 선택

      노드(Node)와 관계(Relationship)로 지식을 표현하는 그래프 데이터베이스입니다. OWL보다 유연하고, 실시간 질의(Cypher)에 최적화되어 있습니다.

      "CNC-001에서 TOOL_WEAR가 발생하면 어떤 부품이 필요한가?"를 그래프 순회 한 번으로 즉시 답할 수 있습니다.

      우리 프로젝트: F4 GraphRAG에서 실시간 질의 필요 → Neo4j 선택

      왜 OWL이 아닌 Neo4j인가? — 우리 시스템은 "센서 이상 감지 → 2~10초 내 LLM이 조치 판단"을 해야 합니다. OWL 추론은 배치 처리에 적합하지만, 실시간 그래프 순회 + Cypher 쿼리는 Neo4j가 압도적으로 빠릅니다. 또한 Neo4j는 별도 추론 엔진 없이도 관계를 타고 탐색하는 것만으로 "이 고장에 필요한 부품 → 현재 재고"를 바로 찾을 수 있습니다.

      GraphRAG란?

      GraphRAG를 이해하려면 먼저 RAG를 알아야 합니다.

      RAG (Retrieval-Augmented Generation)

      LLM한테 질문할 때, 관련 문서를 먼저 찾아서 같이 넘겨주는 기법입니다.

      일반 LLM (RAG 없이)

      "TOOL_WEAR 수리 방법?" → LLM이 일반 지식으로 대답
      문제: 우리 설비에 맞지 않는 답, 환각(hallucination) 위험

      RAG 적용

      "TOOL_WEAR 수리 방법?" → 정비 매뉴얼에서 관련 문서를 먼저 검색 → 찾은 문서를 LLM에게 "이거 참고해서 답해줘"라고 같이 전달 → 근거 있는 답변

      그러면 GraphRAG는?

      RAG의 "검색" 부분에 지식 그래프를 쓰는 것입니다.

      일반 RAG는 문서를 벡터 유사도(pgvector)로 검색합니다. "TOOL_WEAR"와 의미적으로 비슷한 문서를 찾는 거죠. 하지만 벡터 검색만으로는 구조적 관계를 모릅니다:

      • "TOOL_WEAR에 필요한 부품이 뭔지" — 관계를 타야 알 수 있음
      • "과거에 같은 고장을 어떻게 수리했는지" — 정비 이력 순회 필요
      • "그 부품 재고가 지금 있는지" — ERP 조인 필요

      왜 이 데이터로 충분한가? (MVP 관점)

      합성 데이터를 쓰는 게 찜찜할 수 있습니다. 하지만 우리의 목표는 "파이프라인이 동작하는 것을 증명하는 MVP"이지, 상용 배포가 아닙니다.

      이 프로젝트가 증명하려는 것

      센서가 이상 신호를 보내면 → IT 데이터(작업/재고/정비이력)와 합쳐서 → 온톨로지로 관련 지식을 찾고 → LLM이 "무엇을 해야 하는지"를 자율 판단하는 전체 흐름(F1→F6)이 끊기지 않고 작동하는 것입니다.

      SCADA (실데이터) — 왜 충분한가

      미시간대 SMART Lab의 실제 CNC 밀링 실험입니다. 42개 센서, 18개 실험, 100ms 고해상도. 센서 패턴의 다양성(마모/정상, 고속/저속, 중단/완료)이 충분히 확보되어 있어서 F2 이상탐지 모델의 학습/검증이 가능합니다.

      MES/CMMS/ERP (합성) — 왜 충분한가

      기업 IT 데이터는 비공개입니다(ADR-002). 하지만 파이프라인이 필요한 건 "데이터의 존재와 형식"이지 "실제 비즈니스 정확도"가 아닙니다. 합성 데이터로도 F3 동기화, F4 GraphRAG 검색, F5 LLM 판단의 흐름과 인터페이스를 완전히 검증할 수 있습니다.

      핵심 테스트 시나리오: W5(5주차)에 P003 Coolant 재고가 0이 됩니다. 이때 COOLANT_LOW_001 고장이 감지되면, F5 LLM은 "수리하고 싶지만 부품이 없다 → 긴급 발주 필요"라는 판단을 내려야 합니다. 이 시나리오 하나만으로도 SCADA → MES → CMMS → ERP → 온톨로지 → LLM의 전체 파이프라인이 정상 작동하는지 검증할 수 있습니다.

      실제 현장 도입 시에는?

      합성 데이터를 실제 MES/ERP API로 교체하면 됩니다. 파이프라인의 인터페이스(입출력 형식)가 동일하도록 합성 스키마를 설계했기 때문에, 데이터 소스만 바꾸면 코드 변경 없이 실데이터로 전환 가능합니다. 이것이 합성 데이터의 진짜 가치입니다.

      노드 상세 — "무엇"이고 "왜" 필요한가

      01 Equipment — CNC 설비 (3대)

      무엇: CNC 밀링 머신 한 대를 나타내는 노드입니다. CNC-001, CNC-002, CNC-003 세 대가 있습니다.

      왜 필요한가: 모든 질문의 출발점입니다. "CNC-001에 이상이 생겼다" → 이 설비에 달린 센서는? 과거에 겪은 고장은? 현재 작업은? 이 모든 질의가 Equipment 노드에서 시작됩니다.

      equipment_id (PK) equipment_type status experiment_range

      실제 인스턴스: CNC-001 (exp01~06, 신품 공구 중심) / CNC-002 (exp07~12, 마모 빈번) / CNC-003 (exp13~18, 오래된 공구 위주)

      02 Sensor — 센서 메타데이터 (42개)

      무엇: 각 CNC 설비에 달린 센서 하나를 나타냅니다. 원본 48개 중 분석 의미가 있는 42개만 노드로 생성합니다.

      왜 필요한가: "어떤 센서가 이상 신호를 보였는가?"를 추적하기 위해서입니다. 예를 들어 X1_OutputCurrent(X축 출력 전류)가 급증했다면, 이 센서가 어떤 고장코드와 연관되는지 DETECTS 관계를 타고 찾을 수 있습니다.

      왜 42개만? Z축 전기 피드백 4개(all zeros), S1_SystemInertia(상수 12.0), M1_sequence_number(행 순번)는 이상탐지에 쓸 수 없어서 제외합니다. 이 판단은 Phase 1 EDA에서 검증 완료.

      sensor_id (PK) sensor_type (전류/전압/전력/위치/속도/가속도) axis (X1/Y1/Z1/S1/M1) unit

      03 FailureCode — 고장 유형 (4종) ← 그래프의 중심 허브

      무엇: CNC 설비에서 발생할 수 있는 고장의 종류입니다. 4개의 고장코드가 정의되어 있습니다.

      왜 필요한가: 온톨로지의 핵심 허브입니다. 센서가 이상을 감지하면(DETECTS) 고장코드를 거쳐서 → 필요 부품(REQUIRES), 정비 매뉴얼(DESCRIBED_BY), 과거 정비(RESOLVES)로 갈라집니다. FailureCode가 없으면 "이상 신호를 감지했는데 뭘 해야 하지?"에 답할 수 없습니다.

      TOOL_WEAR_001

      공구 마모 — 가장 흔한 고장. 엔드밀 교체 필요

      SPINDLE_OVERHEAT_001

      주축 과열 — 베어링 마모 또는 냉각 부족

      CLAMP_PRESSURE_001

      클램프 압력 이상 — 볼트 풀림 또는 유압 문제

      COOLANT_LOW_001

      절삭유 부족 — 보충 또는 필터 교체

      04 Part — 부품 (5종)

      무엇: 정비에 사용되는 소모품/교체 부품입니다.

      왜 필요한가: LLM이 "엔드밀을 교체하세요"라고 판단했을 때, 재고가 있는지 확인해야 합니다. Part 노드는 FailureCode와 연결(REQUIRES)되어 "이 고장엔 이 부품이 필요"를 알려주고, PG의 erp_inventory와 조인하면 "지금 몇 개 남았는지"까지 답할 수 있습니다.

      05 Document — 정비 매뉴얼 (12건)

      무엇: 고장별 정비 절차, 안전 주의사항, 교체 방법을 담은 문서입니다. 4종 고장코드 × 3종 문서유형(교체 절차서, 점검 체크리스트, 트러블슈팅) = 12건 합성 완료.

      왜 필요한가: LLM이 조치를 판단할 때, "왜 이렇게 해야 하는지"의 근거가 되는 문서입니다. GraphRAG의 RAG(Retrieval-Augmented Generation) 부분이 바로 이 문서를 검색하는 것입니다. 매뉴얼 없이는 LLM이 일반 지식으로만 답하게 되어 환각(hallucination) 위험이 커집니다.

      Neo4j에는 메타데이터(제목, 문서 타입, 관련 고장코드)만 저장하고, 실제 텍스트와 임베딩은 PostgreSQL pgvector에 저장합니다. 검색 시 Neo4j로 관련 문서를 좁히고, pgvector로 정밀 의미 검색을 수행합니다.

      06 WorkOrder — 작업지시 (18건)

      무엇: MES에서 내려온 "이 설비로 이 제품을 가공하라"는 지시입니다.

      왜 필요한가: 이상이 감지되었을 때 "지금 이 설비가 뭘 하고 있는지"를 알아야 합니다. 긴급(urgent) 작업 중이면 즉시 정지 대신 감속 운전을 선택할 수 있고, 납기가 임박하면 정비 일정을 조율해야 합니다.

      priority: normal / urgent / critical · status: completed / aborted

      07 MaintenanceAction — 정비 이벤트 (39건)

      무엇: 실제로 수행된 정비 기록입니다. 교정 정비 12건 + 예방 정비 27건.

      왜 필요한가: "과거에 같은 고장을 어떻게 고쳤는가"의 답입니다. LLM이 조치를 판단할 때, 과거 성공 사례를 참고하면 더 정확한 권고를 할 수 있습니다. 또한 어떤 부품을 소모했는지(CONSUMES), 어떤 매뉴얼을 참조했는지(REFERENCES)도 추적됩니다.

      event_type: corrective / preventive · duration_min: 평균 작업 시간

      관계 상세 — "왜" 이 관계를 맺었는가

      각 관계는 위의 6가지 핵심 질문 중 하나 이상에 답하기 위해 존재합니다.

      R1 HAS_SENSOR Equipment → Sensor · 1:N

      왜: "CNC-001에 달린 센서가 뭐가 있지?" — 이상 발생 시 해당 설비의 센서 목록을 가져와 어떤 센서가 문제인지 범위를 좁힙니다.

      R2 DETECTS Sensor → FailureCode · N:M

      왜: "X1_OutputCurrent가 급증하면 어떤 고장일 가능성이 높은가?" — 센서 이상 패턴을 고장코드와 연결합니다. 속성: anomaly_pattern(이상 패턴 설명), lead_time_min(이상 감지~실제 고장 간 시간). 현재 매핑은 도메인 지식 기반 초기 가설이며, Phase 3에서 모델 결과로 업데이트합니다.

      R3 EXPERIENCES Equipment → FailureCode · N:M

      왜: "CNC-002에서 SPINDLE_OVERHEAT_001이 이전에도 발생한 적 있나?" — 설비별 고장 이력을 추적합니다. 반복 고장이면 근본 원인 분석이 필요하다는 신호입니다.

      R4 REQUIRES FailureCode → Part · N:M

      왜: "TOOL_WEAR_001을 고치려면 엔드밀(P001)이 몇 개 필요한가?" — 고장 → 필요 부품 매핑. 속성: quantity(필요 수량), urgency(긴급도). 이 정보가 있어야 ERP 재고와 대조해서 "부품 있으니 바로 수리 가능" vs "발주 필요"를 판단합니다.

      R5 DESCRIBED_BY FailureCode → Document · N:M

      왜: "SPINDLE_OVERHEAT_001의 정비 절차가 적힌 매뉴얼은?" — LLM에게 근거 문서를 제공하여 환각을 방지합니다. 속성: relevance_score(관련도). 이게 GraphRAG의 "Graph" 부분입니다.

      R6 EXECUTES Equipment → WorkOrder · 1:N

      왜: "CNC-001이 지금 어떤 작업을 하고 있는가?" — 이상 감지 시 현재 작업의 우선순위/납기를 확인해서 즉시정지 vs 감속운전을 판단합니다.

      R7 TRIGGERS WorkOrder → MaintenanceAction · 1:N

      왜: "이 작업 중에 정비가 필요했던 적이 있는가?" — 작업과 정비의 인과관계를 추적합니다.

      예외: 예방정비 27건은 작업지시 없이 정기 일정으로 수행 → work_order_id가 비어있어 R7 관계 없음. 교정정비 12건만 연결됩니다.

      R8 RESOLVES MaintenanceAction → FailureCode · N:1

      왜: "과거에 TOOL_WEAR를 어떻게 해결했는가?" — 성공적인 정비 사례를 LLM에게 제공합니다. 속성: resolution_time_min(해결 소요 시간) — "보통 30분이면 고칠 수 있다"는 정보.

      예외: 에어 필터 정기 교체 9건(MT-031~039)은 특정 고장코드 없이 수행 → failure_code가 비어있어 R8 관계 없음. R9(CONSUMES → P005)만 연결됩니다.

      R9 CONSUMES MaintenanceAction → Part · N:M

      왜: "과거 정비에서 실제로 어떤 부품을 몇 개 썼는가?" — R4(이론적 필요량)와 R9(실제 소모량)를 비교하면 더 정확한 부품 준비가 가능합니다.

      R10 REFERENCES MaintenanceAction → Document · N:M

      왜: "과거 정비사가 실제로 참조한 매뉴얼은?" — R5(이론적 관련 문서)와 R10(실제 사용 문서)을 비교하면 매뉴얼 검색 정확도를 높일 수 있습니다.

      F4 GraphRAG — 실제 질의 경로 3가지

      이상 감지 시 F4가 Neo4j에서 수행하는 실제 그래프 순회 경로입니다. 최대 3 hop까지 탐색합니다.

      경로 1 : 고장 → 필요 부품 → 현재 재고

      FailureCode —REQUIRES→ Part —(PG JOIN)→ erp_inventory

      용도: "이 고장을 고치려면 엔드밀이 2개 필요한데, 지금 재고가 3개 있으니 바로 수리 가능합니다."

      경로 2 : 고장 → 과거 정비 → 소모 부품

      FailureCode ←RESOLVES— MaintenanceAction —CONSUMES→ Part

      용도: "과거에 TOOL_WEAR를 3번 수리했는데, 매번 엔드밀 1개 + 에어필터 1개를 교체했습니다."

      경로 3 : 고장 → 정비 매뉴얼 → 의미 검색

      FailureCode —DESCRIBED_BY→ Document —(pgvector)→ 유사 청크 검색

      용도: "SPINDLE_OVERHEAT_001 관련 매뉴얼 3건을 찾고, 그 중 '베어링 교체 절차' 섹션을 LLM에게 전달합니다."

      PostgreSQL(포스트그레스) ↔ Neo4j(네오포제이) — 왜 두 개를 쓰는가?

      한마디로: PostgreSQL은 "숫자를 빠르게 집계"하고, Neo4j는 "관계를 빠르게 탐색"합니다.

      센서 데이터 25,286행을 "최근 30초 평균"으로 집계하는 건 PostgreSQL(TimescaleDB)이 빠르고, "이 고장코드에 필요한 부품 → 그 부품을 과거에 쓴 정비 이력"을 3단계 관계로 찾는 건 Neo4j가 압도적으로 빠릅니다.

      ADR-003 결정: 1인 개발에서 DB 4개는 비현실적 → PostgreSQL(TimescaleDB + pgvector 통합) + Neo4j = 총 2개로 제한

      예: failure_codes 테이블의 TOOL_WEAR_001

      # maintenance_events 테이블 (FK를 가진 쪽)

      Phase 2 : DB 스키마

      온톨로지가 "지식의 구조"를 정의했다면, DB 스키마는 "실제 데이터를 어디에 어떻게 저장할지"를 정의합니다.

      실제 현장에서 데이터는 어떻게 흐르는가?

      실제 공장에서는 SCADA, MES, ERP 각각이 자체 데이터베이스를 갖고 있습니다. 그리고 예지보전 시스템은 이 DB들에서 데이터를 가져와서 자기 전용 DB에 저장한 뒤 분석합니다.

      # 실제 공장의 구조
      SCADA DB (센서 실시간) —─┐
      MES DB   (작업지시)    —─┤—→ [예지보전 시스템 DB] —→ 이상탐지 → LLM 판단
      ERP DB   (부품 재고)    —─┘

      우리 프로젝트도 정확히 이 구조입니다. 다만 SCADA/MES/ERP의 DB가 없으니, CSV 데이터를 시뮬레이터가 읽어서 실제 현장처럼 5초마다 INSERT하는 방식으로 실시간 환경을 재현합니다.

      실제 현장

      SCADA → API/OPC-UA로 센서값 전송
      MES → API로 작업지시 전송
      ERP → API로 재고 조회
      ↓
      예지보전 DB에 INSERT

      우리 프로젝트

      Kaggle CSV → 시뮬레이터가 한 행씩 읽음
      MES CSV → 시뮬레이터가 읽음
      ERP CSV → 시뮬레이터가 읽음
      ↓

      핵심: DB에 INSERT된 이후의 모든 것(F1 전처리 → F2 이상탐지 → F3 동기화 → F4 GraphRAG → F5 LLM → F6 대시보드)은 실제 현장과 완전히 동일한 코드입니다. 바뀌는 건 데이터 소스(CSV 시뮬레이터 vs SCADA API)뿐이고, 파이프라인 자체는 그대로 쓸 수 있습니다.

      그래서 이 DB 스키마는 무엇을 정의하는가?

      "실시간으로 쌓이는 센서 데이터 + IT 이벤트를, 어떤 구조의 테이블에 저장해야 F1~F6 파이프라인이 빠르고 정확하게 작동하는가"를 정의합니다.

      단순히 "CSV를 DB에 옮긴다"가 아니라, 실제 현장의 실시간 환경에서 필요한 것들을 고려한 설계입니다:

      • 5초마다 INSERT되는 센서 데이터를 빠르게 집계하려면? → TimescaleDB Hypertable
      • 여러 기능(F1~F6)이 동시에 읽고 써야 한다면? → DB의 동시 접근 제어
      • "이 고장의 관련 매뉴얼"을 자연어로 검색하려면? → pgvector 벡터 인덱스
      • 오래된 센서 데이터가 디스크를 채우면? → 7일 자동 압축 정책
      • 나중에 실제 SCADA로 바꿀 때 DB 구조를 바꿔야 하나? → 안 바꿔도 됨. INSERT 소스만 바뀜.

      01 sensor_readings — 센서 시계열 (가장 큰 테이블)

      롱(Long) 형식 — 선택 안 함

      시각    | 센서명           | 값

      08:00:00 | X1_DCBusVoltage   | 319
      08:00:00 | Y1_CurrentFeedback | 2.8

      와이드(Wide) 형식 — 우리의 선택

      시각    | X1_Current | X1_DCBus | Y1_Current | ...
      08:00:00 | 3.2        | 319      | 2.8        | ...
      08:00:05 | 3.1        | 320      | 2.7        | ...

      행 수 그대로 25,286행. 빠름.

      PK (기본 키)
      timestamp + equipment_id
      데이터 컬럼
      42개
      총 컬럼
      44개
      자동 압축
      7일 후

      컨텍스트 3개: machining_process(가공 단계), m1_current_program_number(프로그램 번호), m1_current_feedrate(이송속도). 센서가 아니라 가공 상황 메타 정보입니다.

      02 anomaly_scores — F2 이상탐지 결과

      무엇: F2가 "지금 이 설비에 이상이 있는가?"를 판단한 결과입니다.

      왜 sensor_readings에 안 넣고 별도 테이블인가:

      • 생성 주기가 다름 — sensor_readings는 5초마다, anomaly_scores는 윈도우(30초) 단위
      • 모델 버전 관리 — 나중에 모델을 개선하면 같은 시점의 결과를 비교해야 함
      • NULL 방지 — sensor_readings에 넣으면 대부분의 행에 빈 값이 들어감
      # 예시: CNC-001에서 이상 감지됨

      equipment_id: CNC-001
      anomaly_score: 0.87 ← 1에 가까울수록 이상
      is_anomaly: true
      predicted_failure_code: SPINDLE_OVERHEAT_001

      데이터가 실제로 어떻게 흐르는가

      하나의 이상 감지 이벤트가 DB를 어떻게 거치는지 따라가 봅니다:

      1. F1이 센서 값을 읽어 sensor_readings에 INSERT
      2. F2가 sensor_readings를 읽어 이상 점수를 계산 → anomaly_scores에 INSERT

      4. F3이 mes_work_orders + maintenance_events + erp_inventory를 SELECT → 현재 상황 파악

      7. F6가 전체 결과를 대시보드에 표시

      Phase 2 : 파이프라인 & API

      F1~F6이 실제로 어떻게 연결되고, 외부에서는 어떻게 접근하며, 문제가 생기면 어떻게 대응하는지를 정의합니다.

      파이프라인이란?

      파이프라인은 "데이터가 처리되는 순서와 경로"입니다. 수도관(pipe)처럼 데이터가 한 단계씩 흘러가면서 가공됩니다.

      우리 파이프라인은 두 가지 모드로 동작합니다:

      →
      F2
      →
      F6

      센서를 읽고(F1), 이상 여부를 판단하고(F2), 대시보드를 갱신(F6)합니다. 이상이 없으면 여기서 끝나고 5초 후 다시 반복합니다.

      하루 약 17,280 사이클 (86,400초 ÷ 5초). 설비 3대 합산 약 51,840행/일. 대부분의 시간은 이 모드.

      모드 2 — 이상 대응 (이벤트 트리거)

      F3
      →
      F4
      →
      F5
      →
      F6

      F2에서 is_anomaly = true가 나올 때만 실행됩니다. 상황 파악(F3) → 지식 검색(F4) → LLM 판단(F5) → 알람 표시(F6).

      드물게 발생. F5는 2~10초 걸리므로 비동기로 실행하여 대시보드 갱신을 막지 않음.

      왜 두 모드를 나눴는가: 이상이 없을 때도 F3~F5를 매번 실행하면 불필요한 DB 쿼리 + LLM API 호출이 발생합니다. 5초마다 LLM을 호출하면 비용도, 응답 시간도 감당이 안 됩니다. "감시는 계속, 분석은 필요할 때만" 전략입니다.

      API 엔드포인트란?

      API 엔드포인트는 "프로그램끼리 대화하는 주소"입니다. 대시보드(F6)가 센서 데이터를 가져오려면 /api/f6/sensors/CNC-001이라는 주소로 요청하면 됩니다. 마치 웹 주소(URL)처럼요.

      우리 시스템은 FastAPI(Python 웹 프레임워크)로 14개 엔드포인트를 제공합니다. 크게 두 종류입니다:

      • POST — "이 데이터를 처리해줘" (F1 수집, F2 감지, F3 동기화, F4 검색, F5 판단)
      • GET — "이 데이터를 보여줘" (이력 조회, 대시보드 표시, 헬스 체크)

      에러 대응 시나리오 — "이것도 고장나면 어떡하지?"

      예지보전 시스템 자체도 고장날 수 있습니다. 센서가 안 읽히거나, DB가 다운되거나, LLM API가 응답을 안 하면? 각 상황별로 "감지 방법 + 대체 방안(Fallback)"을 미리 정해둡니다.

      E1 — 센서 데이터가 5초 넘게 안 들어옴

      영향: F1→F2가 중단됩니다. 이상 감지를 못 합니다.

      감지: 폴링 타이머에서 타임아웃 체크.

      대응: 마지막 정상값으로 빈칸 채우기(Forward-fill). 3회 연속 실패하면 대시보드에 "센서 연결 끊김" 경고 표시.

      E2 — F2 이상탐지 모델 추론 실패

      영향: 이상 여부를 판단할 수 없습니다.

      감지: 모델 추론 함수의 try-except.

      대응: 입력 데이터 크기를 줄여서 재시도. 실패하면 이전 점수를 유지하고 "모델 오류" 플래그 설정.

      E3 — Neo4j 연결이 끊김

      영향: F4의 1단계(그래프 순회)를 못 합니다. 관련 부품, 과거 정비를 못 찾습니다.

      감지: Neo4j 연결 체크 (연결 풀 상태).

      대응: 2단계(pgvector 의미 검색)만으로 진행. 구조적 검색 없이 의미 검색만 하면 정확도는 떨어지지만 완전 중단보다 나음.

      E4 — LLM API 타임아웃 또는 429(요청 한도 초과)

      영향: F5가 자율 판단을 못 내립니다.

      감지: HTTP 상태 코드 확인 (408 타임아웃, 429 Rate Limit).

      대응: 3회 재시도 (지수 백오프: 1초→2초→4초). 그래도 실패하면 "LLM 판단 불가, 수동 확인 필요" 알람을 대시보드에 표시. 이상 감지 자체(F2)는 계속 작동합니다.

      E5 — PostgreSQL이 다운

      영향: 거의 모든 기능에 영향. 센서 저장, 이상 기록, IT 데이터 조회, 벡터 검색 전부 불가.

      감지: 연결 풀에서 ConnectionError 발생.

      대응: 지수 백오프로 재연결 시도. 대시보드에 "DB offline" 표시. 재연결 성공 시 자동 복구. Docker 컨테이너의 restart 정책으로 자동 재시작도 설정.

      E6 — LLM이 환각(Hallucination)을 일으킴

      영향: 존재하지 않는 고장코드(예: "MOTOR_FAIL_007")나 부품(예: "P099")을 권고할 수 있습니다.

      감지: LLM 응답에서 failure_code와 part_id가 실제 마스터 테이블에 존재하는지 DB 조회로 검증.

      대응: 존재하지 않는 ID가 포함되면 "환각 감지"로 판단하고 재시도(최대 2회). 재시도에도 실패하면 "자동 판단 불확실, 수동 확인 필요"로 에스컬레이션.

      응답 시간 기대치 — "얼마나 빨라야 하는가?"

      우리 시스템에서 가장 중요한 것은 "이상 감지부터 판단까지 사람이 알아채기 전에 끝나는 것"입니다.

      <100ms
      F1 수집 · F3 동기화 · F6 요약
      단순 INSERT/SELECT. DB가 가장 잘하는 일.
      200~500ms
      F2 이상탐지 · F4 GraphRAG
      모델 추론 + 그래프 순회 + 벡터 검색.
      2~10초

      가장 느린 병목. 그래서 비동기 처리.

      전체 파이프라인 (이상 감지 → 판단 완료): 약 3~11초. F5가 병목이지만, 비동기로 실행하므로 대시보드 갱신(F6)은 블로킹되지 않습니다.

      비교: 사람이 이상을 인지하고 매뉴얼 찾고 판단하면 최소 10~30분. 시스템은 최대 11초. 이것이 자동화의 가치입니다.
