Manufacturing AX Agent

CNC 설비의 예지보전 + 온톨로지 + LLM 자율 판단 관제 시스템

배경 및 문제 정의

제조 현장의 설비 데이터(SCADA), 생산 맥락(MES), 자원 상태(ERP), 정비 지식(매뉴얼)이 각각 독립적인 시스템(Silo)에 분산되어 저장 및 관리되고 있습니다.

설비 이상 발생 시, 작업자가 3~4개의 시스템을 수동으로 순회하며 상황을 대조하고 판단해야 합니다. 이로 인해 의사결정이 지연되고, 결과적으로 설비의 평균 수리 시간(MTTR)과 비가동 시간(Downtime)이 불필요하게 증가하여 생산 원가 손실이 발생합니다.

프로젝트 목표

기존의 분절된 IT/OT 데이터를 데이터베이스 스키마 단위에서 통합하고, 센서 수치 예측(Forecasting)과 LLM(대형 언어 모델)을 결합하여 설비 정지 전 선제적 알람과 비즈니스 맥락이 반영된 자율 조치 지시서를 즉각적으로 제공하는 AI 에이전트 시스템을 구축합니다.

프로젝트 범위 (Scope)

본 프로젝트는 데이터가 도착한 이후의 분석/판단/관제 레이어(F1~F6)를 구축하는 것이 목표입니다. 실제 장비에 IoT 센서를 부착하거나, FOCAS/OPC UA 같은 제조사 폐쇄 프로토콜을 연결하는 데이터 수집 레이어는 범위 밖입니다.

범위 안: 센서 데이터 전처리(F1) → 이상탐지/예측(F2) → IT/OT 동기화(F3) → GraphRAG 검색(F4) → LLM 자율 판단(F5) → 대시보드(F6)

범위 밖: IoT 센서 부착, 프로토콜 브릿지(FOCAS, OPC UA), 엣지 게이트웨이, 공장 네트워크 접근

수집된 데이터를 온톨로지 + GraphRAG + LLM으로 자율 판단까지 연결하는 것이 차별화 포인트입니다.

우리가 해결하려는 문제

CNC 공장에서 실제로 겪는 3가지 문제. 이걸 해결하려면 데이터가 필요합니다.

1. 이상탐지

장비 센서가 평소와 달라지고 있는데, 사람이 눈치채기 전에 시스템이 먼저 감지해야 합니다.

예를 들어 X축 전류가 평소 0.08A인데 갑자기 0.3A로 튀면 "뭔가 이상하다"고 자동으로 알려주는 것입니다. 비지도 학습 모델(Isolation Forest, AutoEncoder 등)로 현재 센서 수치의 이상 상태(Anomaly Score)를 판별합니다.

2. 예지보전 (Forecasting)

이상이 감지되면, "앞으로 어떻게 될지" 예측해서 지금 멈춰야 하는지, 감속하면 되는지, 다음 정비 때 하면 되는지 판단해야 합니다.

단순히 "이상하다"를 넘어서, "30분 뒤에 위험해진다" 같은 미래 예측이 핵심입니다. 시계열 예측 딥러닝 모델(LSTM 등)을 통해 미래 센서값을 예측하고, 위험 임계치 돌파 여부를 선제적으로 알립니다.

왜 RUL(잔여수명)이 아닌 Forecasting인가? (ADR-001)

RUL은 "이 부품이 몇 시간 뒤에 고장난다"를 직접 예측하는 방식이지만, 신뢰할 수 있는 RUL 타겟 변수를 합성하기 어렵습니다. 대신 실제 센서값의 미래를 예측하고, 그 예측값이 임계치를 돌파하는지로 "언제 위험해지는가"를 판단하는 Forecasting 방식을 채택했습니다.

3. 지식 구조화 (온톨로지)

"이 고장코드가 뜨면 → 이 부품을 교체하고 → 매뉴얼 3장을 참고해"

이런 정비 지식이 지금은 숙련공 머릿속에만 있습니다. 이걸 그래프(온톨로지)로 구조화해서 AI가 검색하고 활용할 수 있게 해야 합니다. Neo4j에 '설비 → 고장 코드 → 필요 부품 → 조치 매뉴얼'의 관계망을 구축하고, 에러 키워드(BM25)와 증상 설명(Vector Search)을 결합한 하이브리드 검색으로 관련 지식을 찾습니다.

예지보전은 하나의 데이터가 아니다

이 프로젝트에서 "예지보전 데이터"라는 단일 테이블이 따로 있는 것이 아닙니다. 예지보전은 아래 데이터가 합쳐져서 만들어집니다:

예지보전 = 센서 신호 + 이벤트 근거 + 운영 맥락 + 자원 맥락 + 정비 지식

그래서 지금 해야 할 일은 "예지보전 데이터셋 하나를 찾는 것"이 아니라, 어떤 데이터를 어떻게 연결해서 예지보전 판단을 만들지 정리하는 것입니다.

예지보전을 구성하는 5가지 데이터

  • 센서 신호 — 전류, 전압, 전력, 속도, 가속도 → 이상탐지, Forecasting, 위험 신호 포착
  • 이벤트 근거 — 알람 발생, 정비 시점, 부품 교체, 고장 시점 → 센서 패턴의 의미 해석, 전조와 실제 이벤트 연결
  • 운영 맥락 — 현재 작업지시, 납기 긴급도, 생산 잔량 → 지금 멈출지, 감속할지, 계획 정비로 넘길지 판단
  • 자원 맥락 — 부품 재고, 리드타임, 최근 교체 이력 → 실제 조치 가능성 판단
  • 정비 지식 — 고장 코드, 조치 절차, 매뉴얼 문서 → GraphRAG 검색, LLM 조치 리포트 근거

F1. 실시간 시계열 데이터 수집 및 전처리

원시 센서 데이터(CNC Mill)를 수집하여 결측치 및 이상치를 통계적으로 제거합니다. 지정된 윈도우(Sliding Window) 크기로 이동 평균, 분산 등 파생 변수(Feature)를 추출합니다.

  • 입력: CNC 센서 CSV (48컬럼, 100ms 샘플링)
  • 출력: 깨끗한 시계열 데이터 (정규화, 결측 처리, 파생 변수 추가)
  • 데이터: Kaggle CNC Mill (핵심) + Bosch CNC (보조, 진동)
  • 확보 방식: 실제 수집 완료

F2. 사전 위험 예측 및 이상 탐지

비지도 학습 모델로 현재 센서 수치의 이상 상태(Anomaly Score)를 판별합니다. 시계열 예측 딥러닝 모델(LSTM 등)로 미래 센서값을 예측하고 위험 임계치 돌파 여부를 선제적으로 알림(Trigger)합니다.

  • 입력: F1 전처리 출력 + 공구 상태 라벨(worn/unworn)
  • 출력: 이상 점수(Anomaly Score) + 미래 예측값
  • 데이터: F1 출력 + 정비/고장 이벤트 로그(합성)
  • 확보 방식: F1 출력(자동) + 이벤트 로그(Phase 2 합성)

F3. IT/OT 데이터 타임라인 동기화

F2에서 알람이 발생한 정확한 시점을 기준으로 RDBMS에 쿼리를 실행합니다. 해당 시점의 작업 지시 내역 및 납기 중요도(MES), 필수 교체 부품의 현재 창고 재고량(ERP)을 조회하여 단일 JSON 형태로 병합합니다.

  • 입력: F1 출력(OT) + MES 작업지시(IT) + ERP 부품 재고(IT)
  • 출력: 통합 운영 맥락 JSON
  • 데이터: MES(work_order_id, due_date, priority) + ERP(part_id, stock_quantity, lead_time_days)
  • 확보 방식: 합성 생성 (ADR-002)

F4. 지식그래프 연동 하이브리드 검색 (GraphRAG)

Neo4j에 구축된 온톨로지 지식그래프를 순회하여 '설비 → 고장 코드 → 필요 부품 → 조치 매뉴얼'의 관계망을 식별합니다. 에러 키워드(BM25)와 증상 설명(Vector Search)을 결합한 하이브리드 검색으로 관련 정비 지식을 찾습니다.

  • 입력: Neo4j 온톨로지 + 정비 매뉴얼 텍스트
  • 출력: 관련 매뉴얼·부품·고장 이력 검색 결과
  • 데이터: 온톨로지 관계(직접 설계) + 매뉴얼(수집+합성) + 고장코드 체계(직접 설계)
  • 확보 방식: Phase 3에서 구축

F5. LLM 기반 자율 조치 판단

F2(이상 점수) + F3(운영 맥락) + F4(정비 지식)의 결과를 LLM에 넘겨, 비즈니스 맥락이 반영된 조치 보고서를 자동 작성합니다. 별도 데이터 수집 필요 없이, 앞 기능의 출력을 조합합니다.

F6. 대시보드 (관제 화면)

F1~F5의 모든 결과를 한 화면에 시각화합니다. ADR-005에 의해 Phase 1부터 조기 착수하여, 기능이 추가될 때마다 시각적으로 검증합니다.

"유효 42컬럼"의 의미

원래 48개 컬럼 중 6개가 분석에 쓸모없습니다.

상수 컬럼 2개: S1_SystemInertia(항상 12.0), M1_CURRENT_PROGRAM_NUMBER(항상 1.0)

2
필수 센서에 "진동, 온도"가 있었는데 실제로 없음
required-data-summary §3.1에 필수 센서로 "진동, 전류, 온도, 주축 속도"를 적어놨지만, 핵심 데이터(Kaggle Mill)에는 진동과 온도가 없습니다. 대신 전류(CurrentFeedback, OutputCurrent), 전압(DCBusVoltage, OutputVoltage), 전력(OutputPower), 위치, 속도, 가속도가 X/Y/Z/S 축별로 풍부하게 있습니다.
→ 해결 완료: required-data-summary §3.1을 실제 확보 센서 기준으로 전면 재작성. "확보되지 않은 센서" 섹션 별도 추가.
3
timestamp와 equipment_id가 모든 데이터셋에 없음
설계 원칙 1번 "모든 데이터는 equipment_id + timestamp 기준으로 연결"인데, 확보한 4개 데이터셋 전부에 이 두 컬럼이 없습니다. 공개 학술 데이터의 한계로, Kaggle CNC Mill은 sequence 번호(행 순서)와 experiment 번호(실험 ID)만 있습니다.
→ Phase 1 EDA 첫 번째 과제로 등록: sequence × 100ms → datetime / experiment_01 → CNC-001 합성 규칙 설계. open-items #1도 업데이트 완료.
4
보조 데이터 2종이 합성 데이터로 의심됨
Multi-Sensor CNC는 센서값이 완벽한 균등분포를 보이고 마모량에 음수 값이 존재합니다. Milling RUL은 진동이 표준정규분포이고 RUL이 완벽한 선형 감소를 보입니다. 둘 다 실제 센서가 아닌 인위적으로 생성된 데이터일 가능성이 높습니다.
→ 해결 완료: 두 데이터셋을 "참고용"으로 명시. 핵심 분석은 Kaggle Mill 기준으로 진행.

크로스 검증 과정에서 수정된 오류 2건

A
"Z축 주의" 표현이 너무 넓었음
처음 분석에서 "Z축 전체에 주의 필요"라고 적었는데, 검증 결과 Z축 전기 피드백 4컬럼(CurrentFeedback, DCBusVoltage, OutputCurrent, OutputVoltage)만 0값이고, Z축의 위치/속도/가속도 6컬럼은 완전히 정상이었습니다.
→ 수정 완료: "Z축 전기 피드백 4컬럼"으로 범위를 좁혀서 기술.
B
"미완료 실험 = 비정상 중단" 잘못된 해석
처음에 미완료 실험 4건(실험 4, 5, 7, 16)을 "비정상 중단"으로 해석하고 이상탐지에 활용 가능하다고 적었는데, 검증 결과 4건 전부 Machining_Process 컬럼이 "End"로 정상 종료되어 있었습니다. 의도적으로 짧게 설계된 실험이지, 비정상 중단이 아닙니다.
→ 수정 완료: "의도적으로 짧게 종료된 실험"으로 정정.

EDA 결과를 바탕으로 F1 전처리 파이프라인(노이즈 제거, 정규화, Sliding Window 피처 추출)의 구체적인 설계를 시작합니다.
