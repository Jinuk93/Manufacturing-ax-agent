# 고객 데이터 업로드 기반 온톨로지 자동화 전략

**상태:** Vision Draft
**최종 수정일:** 2026-03-09

이 문서는 현재 개인 프로젝트를 넘어,
향후 제조 기업 고객에게 제공하는 솔루션으로 확장할 경우를 가정하여 작성합니다.

즉, 이 문서는 **현재 Phase 0 범위를 확정하는 문서가 아니라**
향후 제품화와 고객 온보딩 방향을 정리한 비전 문서입니다.

핵심 질문은 아래와 같습니다.

- 고객마다 데이터 구조가 다른데, 어떻게 공통 솔루션으로 흡수할 것인가?
- 고객이 ERP, MES, SCADA, 센서 데이터를 업로드하면 얼마나 자동으로 세팅할 수 있는가?
- 온톨로지는 고객별로 새로 만드는가, 아니면 공통 구조를 두고 매핑하는가?
- 완전 자동과 반자동 중 어디까지를 제품 범위로 볼 것인가?

이 문서는 현재 [data-collection-strategy.md](../0-project-definition/data-collection-strategy.md)와는 별개로,
"고객 맞춤형 솔루션 제품화" 관점의 방향성을 정리한 문서입니다.

---

## 1. 결론부터

팔란티어처럼 가고 싶다면,
고객마다 온톨로지를 처음부터 새로 만드는 방식보다
**공통 제조 코어 온톨로지 + 고객별 데이터 매핑 + 사람 검토** 구조로 가는 것이 현실적입니다.

즉,

- 스키마는 공통으로 유지하고
- 고객 데이터는 반자동으로 공통 모델에 매핑하고
- 실제 고객 데이터로 그래프 인스턴스를 자동 생성하고
- 사람이 검토·승인하는 구조

가 가장 현실적입니다.

---

## 2. 왜 완전 자동이 아니라 반자동인가

고객사의 제조 데이터는 이름만 비슷하지 실제 구조는 크게 다릅니다.

예를 들어,

- ERP의 `item_code`가 부품인지 완제품인지 불명확할 수 있음
- MES의 `line_id`가 생산 라인인지 설비군인지 고객사마다 다름
- SCADA의 설비명과 ERP의 설비 코드가 서로 다를 수 있음
- 문서에는 같은 부품이 다른 용어로 반복될 수 있음

이런 이유로, 완전 자동 생성은 초기 데모는 가능해도 실제 운영에서는 오류 누적 위험이 큽니다.

따라서 제품 메시지는
"완전 자동 온톨로지 생성"보다
**"자동 추천과 빠른 온보딩"** 이 더 현실적이고 강합니다.

---

## 3. 제품 관점의 핵심 원칙

### 3.1. 공통 Canonical Data Model을 먼저 둔다

고객 원본 데이터를 바로 온톨로지에 넣지 않습니다.
먼저 우리 솔루션의 공통 표준 데이터 모델로 정규화해야 합니다.

이유는 아래와 같습니다.

- 고객마다 컬럼명이 다름
- 고객마다 코드 체계가 다름
- 고객마다 시간 단위와 조인 키가 다름
- 고객마다 설비, 부품, 공정 용어가 다름

즉,
원본 데이터 -> 공통 표준 모델 -> 온톨로지/ML/LLM
구조가 되어야 합니다.

### 3.2. 온톨로지는 두 층으로 나눈다

온톨로지는 아래 두 층으로 구분하는 것이 좋습니다.

1. 스키마 온톨로지
무슨 개체와 관계가 있는지 정의하는 공통 구조

2. 인스턴스 그래프
고객 데이터로부터 생성되는 실제 Equipment, WorkOrder, Part, Alarm 등

자동화의 핵심은 보통 인스턴스 그래프 생성입니다.
스키마 자체를 고객마다 자동 생성하는 것은 비추천입니다.

### 3.3. Human-in-the-loop를 전제로 한다

고객 데이터 자동 매핑 결과를 사람이 검토할 수 있어야 합니다.

필요한 검토 항목 예시:

- 컬럼 매핑 승인
- 설비명 병합 승인
- 부품과 제품 구분 수정
- 문서에서 추출된 고장/증상 관계 승인
- 새 고장 유형 추가 승인

자동화 수준이 높을수록 검토 UI의 중요성도 같이 올라갑니다.

---

## 4. 어떤 형태의 솔루션이 되어야 하나

이 솔루션은 단순한 AI 챗봇이 아니라,
고객사의 운영 데이터를 받아 공통 제조 운영 모델로 바꾸고,
그 위에서 온톨로지, 이상탐지, 예지보전, LLM을 작동시키는 플랫폼에 가깝습니다.

즉 구조적으로는 아래와 같습니다.

고객 원본 데이터
-> 데이터 업로드 및 연결
-> 자동 프로파일링
-> 공통 모델 매핑 추천
-> 온톨로지 인스턴스 생성
-> 문서 기반 지식 보강
-> 이상탐지 / 예지보전
-> LLM 조치 추천

## 5. 향후 코어 온톨로지 방향 예시

아래 내용은 현재 프로젝트에서 확정된 스키마가 아니라,
향후 제품화 단계에서 검토할 수 있는 가설적 방향 예시입니다.

처음부터 범용 제조 전체를 담으려 하지 말고,
제조 운영 전반에서 자주 등장하는 최소 코어 개체를 정하는 것이 좋습니다.

예시 개체:

- Site
- Line
- Equipment
- Sensor
- WorkOrder
- Product
- Part
- FailureMode
- Alarm
- MaintenanceAction
- ManualDocument

예시 관계:

- Line has Equipment
- Equipment has Sensor
- WorkOrder runs on Equipment
- Equipment produces Product
- FailureMode affects Equipment
- FailureMode requires Part
- MaintenanceAction resolves FailureMode
- ManualDocument describes MaintenanceAction

이 코어 모델은 고객 데이터를 어디에 매핑할지 판단하는 기준 후보가 됩니다.

---

## 6. 고객 데이터 업로드 후 자동화 흐름

### 6.1. 1단계: 데이터 업로드

고객은 아래 형태로 데이터를 넣을 수 있어야 합니다.

- CSV
- Excel
- API 연동
- DB 연결
- 파일 드롭 방식

대상 데이터 예시:

- ERP: 품목, 재고, BOM, 구매 리드타임
- MES: 작업지시, 생산실적, 공정 상태
- SCADA/센서: timestamp, equipment_id, sensor_name, value
- 문서: PDF, DOCX, HTML, 작업표준서, 점검표

### 6.2. 2단계: 자동 프로파일링

업로드가 끝나면 자동으로 아래를 분석합니다.

- 컬럼명 추출
- 데이터 타입 추론
- 시간 컬럼 후보 탐지
- ID 컬럼 후보 탐지
- 단위 추정
- 결측률, 중복률 분석
- 시간 간격 분석
- 코드값 분포 분석

이 단계는 "데이터를 이해하는 단계"입니다.

### 6.3. 3단계: 공통 모델 매핑 추천

업로드된 컬럼을 공통 Canonical Model에 매핑하도록 자동 추천합니다.

예시:

- `machine_id` -> Equipment ID 후보
- `wo_no` -> WorkOrder ID 후보
- `item_cd` -> Product ID 또는 Part ID 후보
- `stock_qty` -> Inventory Quantity 후보
- `temp` -> Temperature Sensor 후보

추천 근거는 아래를 함께 사용합니다.

- 컬럼명 유사도
- 데이터 타입
- 값 패턴
- 조인 가능성
- 도메인 사전
- LLM 의미 추론

중요한 것은 자동 확정보다 **신뢰도 점수 기반 추천**입니다.

### 6.4. 4단계: 사용자 검토 및 승인

추천된 매핑을 사용자가 검토하고 수정할 수 있어야 합니다.

예시:

- `item_cd`는 Product가 아니라 Part로 수정
- `line_no`는 실제로 설비군이 아니라 공정라인으로 확정
- `status_code`의 코드값 의미 수동 지정

이 단계를 거쳐야 이후 온톨로지 품질이 올라갑니다.

### 6.5. 5단계: 그래프 인스턴스 자동 생성

검토가 끝나면 공통 스키마 위에 고객 인스턴스 그래프를 생성합니다.

예시:

- Equipment: CNC-001, CNC-002
- Part: BRG-7210C
- WorkOrder: WO-20260301-001
- Alarm: Overheat-20260309-01

즉,
스키마는 공통이고 인스턴스만 고객 데이터로 채워집니다.

### 6.6. 6단계: 문서 기반 온톨로지 보강

고객이 업로드한 매뉴얼과 문서를 읽어 아래 후보를 추출합니다.

- 설비명
- 부품명
- 증상
- 고장 코드
- 조치 절차
- 점검 항목
- 주의사항

이 결과는 바로 확정하지 않고,
기존 그래프와 연결 후보로 제안하는 방식이 적절합니다.

## 7. 향후 제품 모듈 예시

아래 항목은 지금 프로젝트에서 모두 구현해야 하는 범위가 아니라,
향후 제품화 단계에서 필요한 모듈 후보를 정리한 참고 메모입니다.

### 7.1. Ingestion

파일 업로드, API 커넥터, DB 연결

### 7.2. Profiling

컬럼 분석, 결측/중복/타입/시간축 분석

### 7.3. Mapping

공통 모델로 자동 추천 및 사용자 승인

### 7.4. Canonical Store

정규화된 표준 데이터 저장

### 7.5. Ontology Builder

인스턴스 그래프 생성, 관계 연결

### 7.6. Document Intelligence

문서 청킹, 엔터티 추출, 관계 추출

### 7.7. Graph + Search

Neo4j, 하이브리드 검색, GraphRAG

### 7.8. ML Pipeline

이상탐지, forecasting, 피처 생성

### 7.9. Decision Layer

LLM 리포트, 조치 추천, 설명 가능성

### 7.10. Governance

승인 이력, 데이터 계보, 스키마 버전, 감사 추적

## 8. 고객 맞춤형과 제품화의 균형

여기서 가장 중요한 전략 문제는 이것입니다.

- 너무 고객 맞춤형이면 SI 프로젝트가 됨
- 너무 표준화하면 실제 고객 데이터를 흡수하지 못함

따라서 아래 구조가 적절합니다.

- 80: 공통 코어 모델
- 20: 고객별 확장 필드, 코드 매핑, 용어 사전

예를 들면,

- 공통: Equipment, Sensor, WorkOrder, Part, FailureMode
- 고객 확장: 고객 전용 설비군 코드, 내부 상태 코드, 전용 고장 분류

이 방식이 제품성과 고객 맞춤형을 동시에 확보하기 좋습니다.

## 9. 참고용 확장 순서

"업로드하면 온톨로지도 되고 예지보전도 되고 LLM도 다 되는 제품"을 한 번에 만들기보다,
지금 프로젝트가 안정적으로 끝난 이후 아래 순서로 확장을 검토할 수 있습니다.

### 9.1. 1단계

데이터 업로드 + 자동 프로파일링

### 9.2. 2단계

공통 모델 매핑 추천 + 사용자 승인

### 9.3. 3단계

온톨로지 인스턴스 자동 생성

### 9.4. 4단계

문서 업로드 기반 GraphRAG 보강

### 9.5. 5단계

이상탐지

### 9.6. 6단계

예지보전

### 9.7. 7단계

LLM 조치 추천

이 순서가 좋은 이유는,
데이터 구조가 안정되기 전에는 ML과 LLM 품질이 흔들리기 쉽기 때문입니다.

## 10. 현실적으로 자주 부딪히는 문제

### 10.1. 조인 키 부재

SCADA의 설비명과 ERP의 설비 코드가 서로 다를 수 있습니다.

### 10.2. 시간축 불일치

센서는 초 단위인데 ERP는 일 단위일 수 있습니다.

### 10.3. 마스터 데이터 불안정

같은 부품이 이름, 코드, 약어로 여러 번 등장할 수 있습니다.

### 10.4. 문서 품질 문제

스캔 PDF, 표 깨짐, 약어 사용으로 자동 추출이 흔들릴 수 있습니다.

### 10.5. 고객 용어와 표준 용어 차이

예를 들어 주축, 스핀들, 메인 샤프트가 섞일 수 있습니다.

### 10.6. 과도한 자동화 기대

실무에서는 완전 자동보다 human-in-the-loop가 더 중요합니다.

---

## 11. 현재 개인 프로젝트와의 연결점

현재 진행 중인 CNC 단일 도메인 개인 프로젝트는,
장기적으로 위와 같은 제품 구조로 확장할 수 있는 좋은 출발점입니다.

특히 아래가 재사용 가능한 자산이 됩니다.

- CNC 도메인 한정 코어 온톨로지 초안
- 센서 기반 이상탐지 및 forecasting 파이프라인
- MES/ERP 합성 데이터 설계 경험
- GraphRAG를 위한 문서 구조화 경험

즉 지금 프로젝트는 단순 PoC가 아니라,
향후 고객 온보딩형 제조 AI 솔루션으로 확장 가능한 기반 설계 실험이라고 볼 수 있습니다.

---

## 12. 결론

가장 현실적인 방향은 아래 한 줄로 정리할 수 있습니다.

**고객마다 온톨로지를 새로 만드는 시스템이 아니라,
공통 제조 코어 온톨로지에 고객 데이터를 반자동 매핑해 인스턴스 그래프를 자동 생성하는 플랫폼으로 설계한다.**

이 문서는 향후 제품 아키텍처, 고객 온보딩 플로우, 코어 온톨로지 초안 문서의 상위 방향 문서로 활용할 수 있습니다.
