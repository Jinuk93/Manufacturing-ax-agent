# 제품 요구사항 정의서 (PRD)

**프로젝트명:** 예측 기반 자율 조치 에이전트 관제 시스템 (`manufacturing-ax-agent`)
**문서 버전:** v1.0
**최종 수정일:** 2026-03-15
**상태:** 확정 (Phase 0 완료, Phase 1 EDA 반영)

---

## 1. 프로젝트 개요 (Project Overview)

### 1.1. 배경 및 문제 정의 (Problem Statement)

**현상:**
제조 현장의 설비 데이터(SCADA), 생산 맥락(MES), 자원 상태(ERP), 정비 지식(매뉴얼)이
각각 독립적인 시스템(Silo)에 분산되어 저장 및 관리됨.

**비즈니스 문제:**
설비 이상 발생 시, 작업자가 3~4개의 시스템을 수동으로 순회하며 상황을 대조하고 판단해야 함.
이로 인해 의사결정이 지연되고, 결과적으로 설비의 평균 수리 시간(MTTR)과
비가동 시간(Downtime)이 불필요하게 증가하여 생산 원가 손실이 발생함.

### 1.2. 프로젝트 목표 (Objective)

기존의 분절된 IT/OT 데이터를 데이터베이스 스키마 단위에서 통합하고,
**센서 수치 예측(Forecasting)**과 LLM(대형 언어 모델)을 결합하여
**설비 정지 전 선제적 알람과 비즈니스 맥락이 반영된 자율 조치 지시서를
즉각적으로 제공하는 AI 에이전트 시스템**을 구축함.

## 2. 핵심 기능 요구사항 (Functional Requirements)

데이터가 유입되어 최종 화면에 출력되기까지의 순차적 파이프라인에 따라
6대 핵심 기능을 정의합니다.

### F1. 실시간 시계열 데이터 수집 및 전처리

- 현재 확보한 **Kaggle CNC Mill Tool Wear 원시 센서 데이터(100ms 샘플링, experiment 단위)**를 기준으로 결측치 및 이상치를 통계적으로 제거
- Phase 1에서 `sequence -> timestamp`, `experiment -> equipment_id` 규칙을 정의하여 내부 표준 시계열 형식으로 정규화 (✅ 합성 완료)
- 지정된 윈도우(Sliding Window) 크기로 이동 평균, 분산 등 파생 변수(Feature) 추출

### F2. 사전 위험 예측 및 이상 탐지 (Forecasting & Anomaly Detection)

- 비지도 학습 모델을 통해 현재 센서 수치의 이상 상태(Anomaly Score) 판별
- 시계열 예측 딥러닝 모델(LSTM 등)을 통해 미래(예: 30분 뒤)의 위험 임계치 돌파 여부를
  선제적으로 예측하여 알람(Trigger) 발생
- 신뢰성이 떨어지는 가상의 잔여 수명(RUL) 데이터 합성을 배제하고,
  실제 센서 데이터의 미래 수치를 예측하는 방식으로 예지보전 목적 달성 (ADR-001 참조)

### F3. IT/OT 데이터 타임라인 동기화 (Data Alignment)

- F2에서 알람이 발생한 정확한 시점(Timestamp)을 기준으로 RDBMS에 쿼리 실행
- 해당 시점의 작업 지시 내역 및 납기 중요도(MES), 필수 교체 부품의 현재 창고 재고량(ERP)을
  조회하여 단일 JSON 형태로 병합

### F4. 지식그래프 연동 하이브리드 검색 (GraphRAG Retrieval)

- Neo4j에 구축된 온톨로지 지식그래프를 순회하여
  '설비 -> 고장 코드 -> 필요 부품 -> 조치 매뉴얼'의 관계망 식별
- 에러 키워드(BM25)와 증상 설명(Vector Search)을 결합한 하이브리드 검색으로
  가장 연관성 높은 정비 매뉴얼 텍스트 색인

### F5. 컨텍스트 기반 LLM 자율 조치 리포트 생성

- F3(비즈니스 상태)과 F4(정비 지식)에서 수집된 데이터를 통합하여 LLM 프롬프트에 주입
- "현재 생산 중인 제품의 납기와 부품 재고를 고려할 때, 설비를 즉시 정지할지 속도를 늦출지"
  판단하고 구체적인 작업 지시를 텍스트 리포트로 자동 생성
- LLM은 반드시 검색된 컨텍스트 내에서만 답변을 생성하도록 시스템 프롬프트를 강제하며,
  출처 없는 조언은 차단함 (Hallucination 방지)

### F6. 운영자 관제 대시보드 (Dashboard UI)

- 센서의 실시간 흐름, 예측된 위험 임계치, 그리고 LLM이 생성한 조치 리포트를
  운영자가 시각적으로 확인할 수 있는 웹 프론트엔드 화면 제공

## 3. 시스템 및 인프라 요구사항 (System Requirements)

개인 프로젝트의 리소스 한계를 고려하여,
복잡도를 최소화한 모놀리식(Monolithic) 및 통합 데이터베이스 아키텍처를 채택합니다.

### 3.1. 기술 스택 (Tech Stack)

| 영역 | 기술 | 비고 |
|------|------|------|
| 데이터베이스 (통합) | PostgreSQL + TimescaleDB + pgvector | 단일 DB로 시계열/관계형/벡터 처리 (ADR-003) |
| 그래프 데이터베이스 | Neo4j | 온톨로지 구조 저장 (대체 불가) |
| 백엔드 API / 모델 서빙 | Python, FastAPI | |
| 프론트엔드 | React 18 + Vite + TanStack Query + Recharts | 내부 관제 SPA (ADR-006) |
| 인프라 | Docker Compose | 컨테이너 통합 실행 환경 |

### 3.2. 비기능 요구사항 (Non-Functional)

- **실시간성 모사:** 과도한 인프라(Kafka 등) 구축을 피하기 위해, 데이터베이스에 배치(Batch) 적재 후
  백엔드와 프론트엔드가 5초 단위로 폴링(Polling)하여 실시간 스트리밍 환경을 모사함 (ADR-004)
- **환각(Hallucination) 방지:** LLM은 반드시 검색된 컨텍스트 내에서만
  답변을 생성하도록 시스템 프롬프트를 강제하며, 출처 없는 조언은 차단함

## 4. 제약 사항 (Constraints)

### 4.1. 도메인 제약

모든 시스템의 데이터 모델링 기준은 **'CNC 기계 가공 공장'** 단일 도메인으로 통일하여
데이터 파편화를 방지함. (ADR-000 참조)

### 4.2. 데이터 제약

- SCADA 센서 데이터: **Kaggle CNC Mill Tool Wear를 핵심 실데이터로 사용**, Bosch CNC는 진동 보조 데이터로 참고
- ~~KAMP 데이터~~: 카탈로그/메타데이터만이었으므로 Phase 1에서 삭제 (Kaggle CNC Mill로 대체)
- MES(작업 지시) 및 ERP(부품 재고): CNC 도메인에 맞춰 Python 스크립트로 합성 완료 (ADR-002, Phase 1)
- 정비/고장 이벤트 로그: failure_code 4종 기반으로 합성 완료 (Phase 1)
- 정비 매뉴얼: 4종 고장코드 × 3종 문서유형 = 12건 합성 완료 (Phase 3 Decided)

### 4.3. 인프라 제약

- 1인 개발 환경으로 DB는 최대 2개(PostgreSQL + Neo4j)로 제한 (ADR-003)
- 실시간 스트리밍 인프라(Kafka, Flink 등) 미사용 (ADR-004)
