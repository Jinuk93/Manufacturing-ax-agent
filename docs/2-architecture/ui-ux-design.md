## 1. 사용자 분석 (Who are we designing for?)

관제 시스템을 실제로 사용하는 사람이 누구인지 먼저 정의했다.
사용자가 달라지면 정보 배치와 우선순위가 완전히 달라지기 때문이다.

| 사용자 | 업무 상황 | 핵심 질문 | 화면에서 가장 먼저 봐야 하는 것 |
|--------|----------|-----------|-------------------------------|
| 현장 작업자 / 기술자 | 설비 옆에서 몸으로 일하면서 화면을 흘끗 봄. 장갑 낀 손으로 터치 가능성 | **지금 뭐가 문제인가?** | 설비 상태 색상 (빨강/초록) |
| 정비 엔지니어 | 알람 받고 대응 판단. 무엇을 어떻게 고쳐야 하는지 즉시 파악 필요 | **무엇을 어떻게 고쳐야 하나?** | LLM 조치 단계 + 필요 부품 |
| 생산 관리자 | 전체 라인 흐름 모니터링. 납기 리스크 판단 | **3대가 다 돌아가고 있나?** | 설비 3대 한눈에 + 작업 납기 |

**설계 원칙으로 도출:**
- 가장 급한 정보(Critical 알람)가 화면 진입 즉시 보여야 한다
- 텍스트보다 색상과 배지로 상태를 먼저 전달한다
- 상세 정보는 한 단계 더 들어가서 볼 수 있도록 계층 구성

## 2. 초안 목업 (v1) 의 문제점

첫 번째 목업은 **Overview 페이지 → 설비 상세 페이지** 구조(2-Page SPA)였다.
검토 결과 아래 문제가 발견되었다.

### 2-1. 페이지 이동 방식의 한계

```
Overview (설비 카드 3개)
  └─ 클릭 → 상세 페이지 (전체 교체)
      └─ 뒤로가기 → Overview
```

| 문제 | 설명 |
|------|------|
| 맥락 손실 | 상세 페이지로 이동하면 다른 2대의 상태가 안 보임 |
| 인지 부하 | 현장에서 "뒤로가기" 버튼을 찾아 누르는 행위 자체가 불필요한 부담 |
| 비교 불가 | CNC-001과 CNC-002를 동시에 비교할 방법이 없음 |

### 2-2. 기능 누락 목록

v1 목업에서 설계는 되어 있었으나 화면에 반영되지 않은 기능들:

| 누락 기능 | 연결 API | 중요도 |
|-----------|---------|--------|
| F4 GraphRAG 참조 문서 (DOC-004 등) | `/api/f4/search` | High — LLM 판단 근거 가시화 |
| F2 이상 점수 히스토리 차트 | `/api/f2/history` | High — "언제부터 올라갔나?" |
| F1~F5 파이프라인 상태 | `/api/health` 확장 | Medium — 백엔드 정상 동작 확인 |
| STOP/REDUCE 임계치 선 | 클라이언트 상수 | High — 숫자보다 선이 직관적 |
| 정비 이력 타임라인 | `/api/f6/maintenance/:id` | Medium — 과거 패턴 맥락 |
| 알람 확인(Acknowledge) 버튼 | `POST /api/f6/alarms/:id/acknowledge` | Medium — 운영 감사 추적 |
| 다중 KPI 카드 | `/api/f6/summary` | Medium — 핵심 지표 한눈에 |

## 3. UX 패턴 선택: 3-Pane Layout

### 3-1. 선택지 비교

| 방식 | 장점 | 단점 | 채택 |
|------|------|------|------|
| **2-Page SPA** (v1) | 구현 단순 | 맥락 손실, 뒤로가기 필요 | ❌ |
| **탭 방식** | 같은 페이지 유지 | 탭 전환 시 여전히 맥락 분리 | ❌ |
| **3-Pane Layout** | 페이지 이동 없음, 모든 맥락 동시 표시 | 화면 폭 필요 (1280px+) | ✅ |
| **Overlay / Drawer** | 메인 화면 유지 | 드로어가 내용 가림 | 보조 용도로만 |

### 3-2. 3-Pane 구조

```
┌──────────────┬──────────────────────────────┬────────────────────────┐
│  LEFT 240px  │  CENTER (flex-1)             │  RIGHT 360px           │
│              │                              │                        │
│  설비 목록   │  선택된 설비의 주요 데이터   │  AI 조치 + 근거        │
│  ──────────  │                              │                        │
│  ● CNC-001  │  [Critical 배너]             │  [이상 점수 게이지]    │
│  🔴 CNC-002  │  [KPI 카드 4개]              │  [STOP/REDUCE/MONITOR] │
│  ● CNC-003  │  [실시간 센서 차트]          │  [조치 단계]           │
│              │  [이상 점수 히스토리]        │  [GraphRAG 문서 3건]   │
│  ──────────  │  [정비 이력 타임라인]        │  [필요 부품 + 재고]    │
│  알람 피드   │                              │  [작업 현황]           │
│  ──────────  │                              │                        │
│  파이프라인  │                              │                        │
│  F1●F2●F3●  │                              │                        │
│  F4●F5●F6●  │                              │                        │
└──────────────┴──────────────────────────────┴────────────────────────┘
```

### 3-3. 인터랙션 흐름

```
화면 진입
  → 가장 Critical한 설비(CNC-002)가 자동 선택됨
  → Center: CNC-002 센서/이상 데이터 표시
  → Right: LLM 조치 리포트 표시
  → 왼쪽 리스트에서 다른 설비 클릭
      → 페이지 이동 없이 Center + Right만 교체
      → Left 목록은 유지 (비교 맥락 살아있음)
```

## 4. 디자인 시스템: Palantir Blueprint

### 4-1. 선택 근거

| 기준 | 이유 |
|------|------|
| 정보 밀도 | 제조 관제는 소비자 앱이 아님. 전문가용 고밀도 UI가 맞음 |
| 다크 테마 | 공장 현장 환경 특성. 장시간 모니터링에 눈 부담 낮음 |

| 특징 | 구현 방법 |

## 5. 정보 계층 설계 (Information Hierarchy)

사용자가 화면을 봤을 때 어떤 순서로 정보를 인식하는지 설계했다.

### 5-1. Left Panel — "어떤 설비가 문제인가?"

```
우선순위:
  1. 빨간 점 + "위험" 배지 → 즉시 시선 포착
  2. anomaly_score 수치
  3. 설비명 (CNC-001/002/003)
  4. 알람 피드 (최근 이상 목록)
  5. 파이프라인 상태 (F1~F6 도트)
```

**자동 포커스:** 페이지 진입 시 가장 높은 anomaly_score 설비가 자동 선택됨.
Critical 설비가 없으면 score 높은 순으로 선택.

### 5-2. Center Panel — "무슨 일이 일어나고 있는가?"

```
우선순위:
  1. Critical 배너 (score ≥ 0.8일 때만 표시) — 화면 최상단
  2. KPI 카드 4개 (이상점수, 고장코드, 확신도, 다운타임)
  3. 실시간 센서 차트 (이상 구간 빨간 하이라이트)
  4. 이상 점수 히스토리 (언제부터 올라갔는지)
  5. 정비 이력 타임라인 (과거에도 이런 일이 있었는지)
```

### 5-3. Right Panel — "무엇을 해야 하는가?"

```
우선순위:
  1. 이상 점수 게이지 (현재 상태 한눈에)
  2. STOP/REDUCE/MONITOR 배너 (행동 지시)
  3. 조치 단계 (번호 리스트, 즉시 실행 가능)
  4. 판단 근거 (왜 이 조치인가)
  5. GraphRAG 참조 문서 (어떤 매뉴얼을 참고했는가)
  6. 필요 부품 + 재고 현황
  7. 작업 현황 (납기 영향 판단)
```

## 6. 알람 처리 UX

### 6-1. 심각도 3단계

| 레벨 | 조건 | 시각 표현 | 사운드 |
|------|------|-----------|--------|
| **CRITICAL** | anomaly_score ≥ 0.8 | 빨간 배너 (전체 폭) + 도트 깜빡임 | Phase 3 결정 |
| **WARNING** | 0.6 ≤ score < 0.8 | 주황 배지 + 알람 피드 | 없음 |
| **MONITOR** | 0.5 ≤ score < 0.6 | 노란 배지 (알람 피드만) | 없음 |

### 6-2. Critical 배너 UX

- 화면 최상단 고정 (스크롤해도 보임)
- "확인 처리" 버튼 → `POST /api/f6/alarms/:id/acknowledge` 호출
- 확인 후 배너 색이 흐려지고 "확인됨 ✓" 표시
- **Phase 3 구현:** 확인자 + 시각 기록 (Audit trail)

### 6-3. 알람 자동 선택 로직

```
페이지 진입 시:
  if (CRITICAL 설비 있음):
    → 해당 설비 자동 선택 + Critical 배너 표시
  else if (WARNING 설비 있음):
    → score 가장 높은 설비 자동 선택
  else:
    → CNC-001 기본 선택
```

## 8. GraphRAG 결과 표시 설계

GraphRAG(F4)는 LLM이 왜 이 판단을 했는지 근거를 가시화하는 핵심 기능이다.
단순히 "AI가 말했다"가 아니라 "어떤 매뉴얼의 어떤 내용을 참고해서 판단했다"를 보여준다.

### 8-1. 표시 정보

```
┌──────────────────────────────────────────┐
│  DOC-004          hybrid score: 0.94     │
│  SPINDLE_OVERHEAT_001 교체절차서          │
│  "스핀들 과열 발생 시 즉시 전원 차단 후   │
│   냉각 대기. 베어링 마모 확인 후 P002…"  │
└──────────────────────────────────────────┘
```

- `manual_id` — 문서 식별자 (DOC-004)
- `title` — 문서 제목
- `hybrid_score` — BM25 + Vector 혼합 점수
- `snippet` — 관련 청크 텍스트 200자 (api-design.md에 필드 추가됨)

### 8-2. API 변경 사항

`RelatedDocument` Pydantic 모델에 `snippet` 필드 추가:
```python
class RelatedDocument(BaseModel):
    manual_id: str
    title: str
    hybrid_score: float
    bm25_score: float
    vector_score: float
    text_preview: Optional[str] = None
    snippet: Optional[str] = None   # UI 표시용 (최대 200자)
```

## 10. 판단 투명성 패널 (AI Decision Transparency)

### 10-1. 왜 필요한가

현장 기술자가 AI 조치를 따르려면 **"왜 이렇게 판단했는지"** 를 납득해야 한다.
현재 F5 `reasoning` 필드는 1~2줄 요약이라 근거가 불충분하다.

> "STOP이라고? 정말? 왜?" → 납득 못 하면 → 무시 → AI 시스템 신뢰 붕괴

### 10-2. 표시할 정보

```
┌── AI 판단 투명성 ──────────────────────────── [펼치기 ▼] ──┐
│                                                            │
│  입력 컨텍스트 (F3)                                        │
│  · 작업지시: WO-2024-007 (urgent, 납기 2시간 남음)         │
│  · 재고: P002 베어링 2개 (경고 수준)                       │
│  · 최근 정비: 2주 전 동일 고장 (72분 소요)                  │
│                                                            │
│  참조 문서 (F4 GraphRAG)                                   │
│  · DOC-004 교체절차서 (hybrid: 0.94)                       │
│  · DOC-006 트러블슈팅가이드 (hybrid: 0.87)                 │
│                                                            │
│  검토한 대안                                               │
│  · REDUCE 검토 → score 0.87로 STOP 기준 초과, 배제         │
│  · MONITOR 검토 → 납기 내 정비 가능하나 위험도 과다, 배제   │
│                                                            │
│  전체 판단 과정                                            │
│  anomaly_score가 STOP 임계치(0.80)를 초과하고, 과거        │
│  동일 고장 사례에서 방치 시 평균 200분 추가 손실 발생.      │
│  urgent 작업 납기까지 2시간 여유가 있어 즉시 정지 후        │
│  75분 정비가 납기 리스크보다 낮다고 판단...                 │
└────────────────────────────────────────────────────────────┘
```

### 10-3. UX 처리

| 항목 | 결정 |
|------|------|
| 기본 상태 | 접혀 있음 (요약 1줄만 표시) |
| 펼치기 | 클릭 시 전체 투명성 패널 확장 |
| 위치 | Right panel — AI 조치 리포트 바로 아래 |
| 구현 | `LLMActionResponse.full_reasoning`, `alternatives_considered` 필드 표시 |

### 10-4. 백엔드 연동

`LLMActionResponse` 모델 확장 (api-design.md 반영 완료):
```python
input_summary: Optional[Dict]         # F3 컨텍스트 요약
rag_documents: Optional[List[str]]    # F4 문서 ID 목록
alternatives_considered: Optional[str] # 대안 검토 설명
full_reasoning: Optional[str]          # 전체 판단 과정
```
→ **LLM 프롬프트에 "대안도 설명해줘" 지시 추가만으로 구현** (F5 개발 시 같이 적용)

## 11. 챗봇 패널 (Interactive AI Assistant)

### 11-1. 왜 필요한가

F5는 **자동 트리거** (이상 감지 → 자동 판단) 이지만,
기술자가 능동적으로 AI에게 질문하고 싶은 상황이 있다.

| 상황 | 질문 예시 |
|------|-----------|
| 조치 전 확인 | "P002 베어링 교체 정확한 순서 알려줘" |
| 원인 파악 | "왜 CNC-002에서 스핀들 과열이 반복돼?" |
| 이력 조회 | "이 설비에서 SPINDLE_OVERHEAT_001이 몇 번 발생했어?" |
| 납기 영향 | "지금 정지하면 WO-2024-007 납기 영향 있어?" |
| 매뉴얼 검색 | "냉각수 보충 방법 알려줘" |

### 11-2. 아키텍처

```
사용자 입력 → POST /api/chat
  ├─ equipment_id 있으면 F3 컨텍스트 로드 (현재 설비 상태)
  ├─ 질문으로 F4 GraphRAG 검색 (관련 문서/이력)
  └─ LLM 호출 (F5와 동일 구조, 트리거만 다름)
       └─ 답변 + 참조 문서 반환
```

**F5와의 차이점:**

| 항목 | F5 (자율 판단) | 챗봇 |
|------|--------------|------|
| 트리거 | 이상 감지 (자동) | 사용자 입력 (수동) |
| 출력 형식 | 구조화된 JSON (STOP/REDUCE/조치단계) | 자연어 답변 |
| F4 사용 | ✅ | ✅ |
| F3 사용 | ✅ | ✅ (선택적) |
| 추가 개발량 | — | 엔드포인트 1개 + 프롬프트 조정 |

### 11-3. UI 레이아웃

```
Right Panel 최하단 고정:

┌── AI에게 질문 ─────────────────────────────────────────────┐
│                                                            │
│  [AI]  CNC-002의 anomaly_score가 0.87로 STOP 임계치를      │
│        초과했습니다. DOC-004 절차에 따라 즉시 정지 후      │
│        P002 베어링 교체를 권장합니다.                       │
│                                                            │
│  [나]  P002 베어링 교체 순서 알려줘                         │
│                                                            │
│  [AI]  DOC-004 3절 기준:                                   │
│        1. feedrate 0 → 2. 스핀들 정지 확인 →               │
│        3. 커버 해제 → 4. 베어링 탈거 (토크렌치 25Nm) →    │
│        5. P002 신품 장착 → 6. 테스트 가동…                 │
│        📄 DOC-004 섹션3 참조                               │
│                                                            │
│  ┌────────────────────────────────────── [전송 ▶] ─┐      │
│  │ 질문을 입력하세요...                              │      │
│  └──────────────────────────────────────────────────┘      │
└────────────────────────────────────────────────────────────┘
```

### 11-4. UX 세부 결정

| 항목 | 결정 |
|------|------|
| 컨텍스트 자동 주입 | 현재 선택된 설비 상태가 자동으로 포함됨 |
| 참조 문서 표시 | 답변 하단에 `📄 DOC-XXX` 링크 |
| 히스토리 | 세션 내 대화 유지 (새로고침 시 초기화) |
| 로딩 표시 | 답변 생성 중 "AI가 생각 중..." 점 애니메이션 |
| 입력 제한 | Enter 전송, Shift+Enter 줄바꿈 |
| 구현 시점 | F5 완성 후 (Phase 3 후반) |
